# 第5章 智能体核心模块与实践

## 5.1 大语言模型调用：从基础到实践

### 5.1.1 大语言模型的定义与作用

大语言模型是近年来人工智能在语言交互与通用人工智能领域发展的核心引擎，它通过在海量文本数据上的预训练，学习到强大的语言理解与生成能力。对于智能体而言，大语言模型是语言理解与推理的核心：它能够将用户输入转化为结构化的任务表示，并在需要时生成可执行的操作指令。大语言模型的作用主要体现在以下三个方面。

- 自然语言理解：解析复杂语义，识别用户语言的真实含义。
- 逻辑推理与知识调用：结合背景知识完成复杂推理。
- 内容生成：根据上下文自动生成文本、代码或工具调用参数。

需要说明的是，从系统架构角度看，大语言模型作为基础能力由应用或智能体系统进行调用；而在智能体运行过程中，大语言模型作为核心决策单元，根据输入内容决定是否触发工具、知识检索或其他模块。

### 5.1.2 调用大语言模型的方式：API调用、SDK封装和框架集成

通常，有三种调用大语言模型的方式。

1. API调用：可通过 RESTful API 发送请求并获取响应，是最常见的方式，适合轻量集成。
2. SDK封装：使用官方或第三方提供的 SDK，在应用中以函数调用的形式快速接入。
3. 框架集成：通过 LangChain、LangGraph 等框架，将大语言模型调用与工具、记忆、规划模块结合，实现更复杂的智能体系统。

为了直观展示，笔者给出了三种大语言模型调用方式的典型 Python 示例。

除了对调用方式进行分类，我们还可以从调用链的角度理解大语言模型的工作流程：首先，用户通过自然语言输入问题；然后，智能体通过 API、SDK 或框架将输入传递给大语言模型；接下来，大语言模型在内部完成语义理解、推理与生成；最后，返回结构化的或自然语言的结果，供应用端展示或执行。表5.1总结了三种调用方式的特点、优缺点和适用场景。总体而言，API 调用适合快速接入与跨语言使用，SDK 封装适合主流应用的日常开发，框架集成则更适合构建复杂智能体系统与产品级应用。研究人员和开发者可根据实际需求与开发成本选择合适的调用方式。

#### API 调用示例

```python
import requests

url = "https://api.***nai.com/v1/chat/completions"
headers = {"Authorization": f"Bearer {your_api_key}"}
data = {
    "model": "gpt-4o",
    "messages": [
        {"role": "user", "content": "解释 LangChain 的作用"}
    ]
}

response = requests.post(url, headers=headers, json=data)
print(response.json()["choices"][0]["message"]["content"])
```

#### SDK 封装示例

```python
from openai import OpenAI

client = OpenAI(api_key="your_api_key")

resp = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "user", "content": "用一句话解释大语言模型的作用"}
    ]

print(resp.choices[0].message.content)
```

#### 框架集成示例

```python
from langchain_openai import ChatOpenAI
from langchain.prompts import ChatPromptTemplate

llm = ChatOpenAI(model="gpt-4o", api_key="your_api_key")

prompt = ChatPromptTemplate.from_messages([
    ("system", "你是一个帮忙写摘要的助手"),
    ("user", "请总结以下文本：{text}")
])

chain = prompt | llm

print(chain.invoke({"text": "LangChain是一个用于构建智能体的框架"}).content)
```

### 5.1.3 基础调用示例：提示工程与常见应用

在大语言模型的实际使用中，提示工程是最重要的技巧之一，它的核心思想是构造合理的输入提示，引导大语言模型按照预期的方式输出结果。通过在系统提示（System Prompt）中设定角色，可以让大语言模型以某种身份进行对话，从而设定其回答的风格和内容。例如，将大语言模型设定为“医生”“法律顾问”或“翻译助手”，就能让回答更贴近特定领域的需求。

除了角色设定，基础的大语言模型调用还包括问答、文本生成、摘要、翻译和工具调用等，下面给几个典型的示例。

#### （1）角色设定

通过在智能体的系统消息指定身份，可以让大语言模型以特定身份回答问题，这种方式常用于虚拟助手、教育场景和模拟对话。

##### 角色设定示例

以下代码示例展示了如何通过智能体系统提示设定模型的角色，使大语言模型以特定身份回答用户问题。

```python
from openai import OpenAI

client = OpenAI(api_key="your_apikey")

resp = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {"role": "system", "content": "你是一名专业医生"},
        {"role": "user", "content": "感冒时应该注意哪些事项？"}
    ]
)

print(resp.choices[0].message.content)
```

#### （2）问答

这是最常见的使用方式，即用户直接提出问题，由大语言模型给出回答。这种方式适用于知识问答及智能客服系统。以下代码演示了基础的问答调用过程。

通过提示词明确指定翻译任务，可以让大语言模型对多种语言进行高质量翻译。相比传统的机器翻译，大语言模型能更好地理解上下文和语境。以下代码演示了大语言模型执行翻译任务的基本调用方式。

#### （6）工具调用（Tool Calling）

大语言模型不仅能生成自然语言，还能根据用户的需求返回结构化数据，用于驱动外部工具或 API，这种方式在构建智能体时尤为重要。以下代码示例展示了大语言模型执行工具调用的基本使用方式，其中最常见的实现形式是函数调用。

##### 工具调用示例

```python
functions = [
    {

        "description": "获取天气",

```

## 5.2 工具调用与模型上下文协议

在智能体系统的运行过程中，工具（Tool）与模型上下文协议（Model Context Protocol，MCP）是两个关键的概念。前者使智能体能够与外部世界交互，后者则提供了一个统一的协议层，确保模型与工具、环境之间的信息传递稳定且高效。二者结合构成了“智能体能动性”的核心——不仅能理解任务，还能真正完成任务。

### 5.2.1 什么是工具

在智能体体系中，工具指的是一类可供模型调用的外部函数、API、数据库查询接口或本地服务。工具的本质是能力扩展，大语言模型本身擅长推理和语言处理，但它并不直接具备访问网络、数据库或本地运算的能力，通过引入工具，智能体可以完成以下任务。

- 搜索引擎调用：实时获取外部最新信息。
- 数据库查询：在企业或科研数据库中进行结构化检索。
- 调用本地函数：执行数学计算、文件读写、信号处理等任务。
- 硬件交互：通过串口控制传感器或执行机器人动作。

换句话说，工具是智能体的“手与脚”，而大语言模型是智能体的“大脑”，两者结合后，能实现闭环的“感知—推理—执行”过程。

### 5.2.2 模型上下文协议简介

MCP 是为智能体生态提出的通用协议标准，它的目标是解决“不同工具如何以统一的方式挂载到智能体中”这一问题。其核心思想是：无论是远程 API、本地函数，还是数据库操作，都可以通过统一的 Schema 进行描述，并以相同的上下文接口暴露给大语言模型。这样，大语言模型就不需要学习每一个工具的细节，而是通过协议自动完成调用。

MCP 的三个关键特征如下。

1. 统一接口：所有工具都以标准化的格式（如 JSON Schema）声明输入与输出。
2. 上下文绑定：工具的可用性、调用频率、状态信息会随上下文动态传递。
3. 可扩展性：开发者可以方便地新增、卸载或替换工具，无须修改模型本身。

可以把 MCP 看作智能体的“插件协议”，让模型与工具之间解耦，具备更强的通用性。

### 5.2.3 如何将工具挂载到智能体中

将工具挂载到智能体中的过程通常包括以下几个步骤。

1. 工具注册：开发者提供工具的描述文件（如函数名、输入参数、返回类型等）。
2. 协议对接：通过 MCP 将工具描述转化为智能体可以识别的格式。
3. 上下文注入：在对话或任务执行过程中，系统会将工具能力注入大语言模型的上下文，使其在规划时能感知到这些可调用资源。
4. 调用与执行：当大语言模型认为某个任务需要调用工具时，它会生成符合 MCP 的调用指令，由执行引擎路由到对应工具，执行后再将结果返回模型。

这样，智能体就不再是个“孤立的回答机器”，而是真正的“行动体”。

#### 1. 构建 MCP Server

为了帮助读者更直观地理解 MCP 的工作方式，本节给出一个最小可运行的服务端示例。开发者在服务端只需完成两步：先创建一个 MCP 实例，再通过 `@mcp.tool()` 装饰器注册若干工具函数。在下例中，笔者定义了三类典型工具：本地加法、网络搜索和读取文件。

这段代码完整地体现了 MCP Server 的基本构建逻辑。

- 实例初始化：通过 `FastMCP("mcp-demo")` 创建服务对象，用于统一管理工具注册和运行。
- 本地计算工具：函数 `add` 展示了最简单的数学运算，它没有任何外部依赖，适合作为“入门级”示例。
- 网络搜索工具：函数 `search_http` 封装了对 SerpAPI 的调用，服务端在代码中读取并配置了 API Key，因此具备访问外部搜索服务的权限。这样，服务端即可通过 MCP 将这一外部搜索能力暴露给客户端，实现“客户端通过 MCP 调用外部 API”。
- 文件读取工具：函数 `read_file` 从 `sample_data` 目录读取文件，体现了对本地数据的访问能力。
- 服务启动：通过 `mcp.run()` 启动服务，客户端可通过 MCP 远程调用以上工具。

从整体来看，这个示例囊括了 MCP Server 最常见的三种场景：本地计算、外部 API 调用，以及本地文件访问。读者可以先在本地运行这段代码，体会 MCP 如何把不同类型的工具统一在同一个服务中。掌握了这个技术点之后，再扩展更多工具或引入更复杂的逻辑会更容易。

##### 服务端示例（server.py）

```python
from pathlib import Path
import os
import httpx
from mcp.server.fastmcp import FastMCP

mcp = FastMCP("mcp-demo")


# 工具1：本地加法
@mcp.tool()
def add(a: float, b: float) -> float:
    return a + b


# 工具2：网络搜索（支持百度、谷歌、必应）
SERPAPI_URL = "https://***papi.com/search"


@mcp.tool()
def search_http(query: str, engine: str = "baidu"):
    """执行网络搜索并返回前5个结果"""
    api_key = os.getenv("SERPAPI_API_KEY")

    if api_key:
        # 使用真实 SerpAPI
        params = {
            "q": query,
            "engine": engine,
            "api_key": api_key,
            "num": 5
        }

        with httpx.Client(timeout=15.0) as client:
            r = client.get(SERPAPI_URL, params=params)
            r.raise_for_status()

        data = r.json()
        results = []

        for item in data.get("organic_results", [])[:5]:
            results.append({
                "title": item.get("title", ""),
                "snippet": item.get("snippet", ""),
                "url": item.get("link", "")
            })

        return results

    return []


# 工具3：读取文件
@mcp.tool()
def read_file(path: str) -> str:
    p = Path("sample_data") / path
    return p.read_text(encoding="utf-8")


if __name__ == "__main__":
    mcp.run()
```

#### 2. 调用 MCP Client

在完成服务端工具的注册与运行后，下一步就是在客户端进行调用。MCP 提供了以下两种方式。

1. 使用官方 SDK 提供的 `demo_client`，通过标准输入/输出（Stdio）协议与服务端通信。Stdio 指客户端和服务端通过进程间的标准输入流（stdin）和标准输出流（stdout）交换消息，无须额外网络连接即可完成双向通信。
2. 使用 `openai_client` 将 MCP 工具注册到 OpenAI 的 Function Call 接口中，使大语言模型能够直接调用工具。

##### 客户端调用示例（demo_client.py）

方式一：Demo Client 调用。在这个示例中，客户端通过 `stdio_client` 启动服务端脚本，并建立 `ClientSession` 会话。建立连接后，它会列出可用工具，并依次调用 `add` 与 `read_file`。

```python
import asyncio
import os
import sys
from pathlib import Path

from mcp import ClientSession, StdioServerParameters
from mcp.client.stdio import stdio_client


async def main():
    server_script = Path(__file__).parent.parent / "src" / "mcp_demo" / "server.py"

    server = StdioServerParameters(
        command=sys.executable,
        args=["-u", str(server_script)]
    )

    async with stdio_client(server) as (read, write):
        async with ClientSession(read, write) as session:
            await session.initialize()

            # 1）列出可用工具
            tools = await session.list_tools()
            print("可用工具：", [t.name for t in tools.tools])

            # 2）调用本地函数 add
            r1 = await session.call_tool("add", {"a": 1.5, "b": 2.25})
            print("add 结果 ->", r1.content)

            # 3）调用 read_file
            r2 = await session.call_tool("read_file", {"path": "hello.txt"})
            print("read_file 结果 ->", r2.content)


if __name__ == "__main__":
    asyncio.run(main())
```

本段代码的逻辑如下。

1. 使用 `StdioServerParameters` 指定服务端脚本路径，并通过子进程的方式启动服务端。
2. 使用 `ClientSession` 建立客户端会话，并调用 `session.initialize()` 完成初始化。
3. 通过 `list_tools()` 列出所有注册的工具。
4. 分别调用 `add` 和 `read_file` 工具，验证服务端功能是否可用。

这一方式最大的优点是简单直观，适合快速测试 MCP 服务是否正常。

##### OpenAI 集成示例（openai_client.py）

方式二：OpenAI Client 调用。如果希望大语言模型具备工具调用能力，那么可以将 MCP 工具直接注册到 OpenAI 的 Function Call 接口中，这样，用户只需用自然语言给出指令，模型即可在需要时自动调用相应的工具。

```python
from openai import OpenAI
from mcp.client.openai import register_mcp_tool

# 初始化 OpenAI
client = OpenAI()

# 将 MCP 工具注册到模型上下文
register_mcp_tool(client, "add")
register_mcp_tool(client, "search_http")
register_mcp_tool(client, "read_file")

# 大语言模型通过 Function Call 触发工具
resp = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[
        {"role": "user", "content": "请帮我读取 hello.txt 的内容"}
    ]
)

print(resp.choices[0].message)
```

本段代码的工作原理如下。

1. 使用 `register_mcp_tool` 将 MCP Server 提供的工具注册到 OpenAI 客户端。
2. 当用户输入自然语言请求时，大语言模型在生成回答的过程中会自动判断是否要调用某个 MCP 工具。
3. 如果大语言模型选择调用具体的工具（如 `read_file`），则会通过 Function Call 机制向 MCP Server 发起调用，最终返回结果给用户。

在这种方式下，推理与执行被统一在一次对话中完成：大语言模型不仅能回答问题，还能在必要时主动调用具体的工具，从而具备更强的执行力与实用性。

综上，`demo_client` 更适合开发者调试与验证工具功能，`openai_client` 则展示了如何在实际应用中，将 MCP 与大语言模型无缝结合，使大语言模型具备自动化工具调用能力。

## 5.3 Agent Skills：能力模块化与执行层抽象

### 5.3.1 提出背景与设计动机

在前文所述的大语言模型调用、工具（Tool）与模型上下文协议（MCP）的基础上，一个关键的工程问题逐渐显现：随着智能体能力规模的不断扩展，如何在保持系统可控性与可维护性的前提下，实现能力的复用、组合与演化。

在原型阶段，常见做法是将推理逻辑与执行逻辑直接耦合在智能体内部。这种方式虽然实现成本较低，但系统复杂度提升后，将带来一系列工程问题，包括模块高度耦合、能力难以复用、测试与调试成本上升，以及不利于教学与工程规范化。

因此，有必要在智能体体系中引入清晰的能力抽象层，对“推理”与“执行”进行结构性分离。在此背景下，智能体技能（Agent Skills）作为一种能力模块化与执行层解耦的设计范式被提出，并逐渐被主流智能体系统采用。

### 5.3.2 Agent Skills 的定义

Agent Skills 指将智能体的可执行能力进行独立封装，使其以模块化、可复用、可组合的形式存在，并由智能体在运行时进行统一调度。其核心思想是明确区分推理、调度与执行三个层级。

在该范式下，各组件的职责划分如下。

- 大语言模型（LLM）：负责自然语言理解与高层推理，将用户输入转化为结构化的中间表示（Intermediate Representation）。
- 智能体（Agent）：负责意图解析、任务规划、技能选择与执行调度。
- 技能（Skill）：负责执行确定性功能，完成具体操作。

需要强调的是，Skill 不参与语义推理，也不对用户意图进行解释，其行为完全由输入参数决定，从而具备良好的可测试性、可复用性与工程稳定性。

### 5.3.3 Agent Skills 分层架构

图5.2给出了 Agent Skills 的典型分层架构示意图。该架构将智能体系统划分为推理层、调度层与执行层，形成清晰的纵向抽象。推理、调度与执行在结构上解耦。

```text
大语言模型
（语义理解、知识推理、结构生成）
        │
        ▼
智能体
（意图解析、任务规划、技能调度）
        │
        ▼
执行层 Skills
├── PPT生成 Skill
├── 搜索/数据 Skill
└── 文件/系统 Skill
```

**图5.2 Agent Skills 的典型分层架构示意图**

### 5.3.4 Skill 声明文件与能力感知

在工程实践中，Skill 不仅包含具体的执行代码，还需要配套的能力声明文件，以支持 Agent 的能力感知与调度。本书采用 `SKILL.md` 文件对 Skill 的语义能力进行显式描述。

具体而言：`SKILL.md` 文件用于声明 Skill 的名称、功能描述、输入和输出约束、确定性属性、副作用等信息，其作用类似于能力契约（Capability Contract）。Agent 在运行时读取该声明文件，以感知系统中可用的执行能力，但并不从中生成具体内容。

需要注意的是，`SKILL.md` 仅用于能力声明与调度依据，不参与内容生成，也不作为提示（Prompt）输入给大语言模型。所有语义理解与任务推理均由 LLM 在推理层完成，而 Skill 被严格限定为仅执行结构化输入所描述的确定性操作，不具备推理或决策能力。

### 5.3.5 Agent Skills 与 Tool / MCP 的关系

在抽象层级上，Agent Skills 并不等同于工具（Tool）或模型上下文协议（MCP），而是对二者的进一步封装与组织。

- Tool 通常表示一个最小可调用单元，如函数、API 或系统接口。
- MCP 提供工具的统一描述、上下文管理与调用协议。
- Skill 通常封装一个或多个 Tool，并配合能力声明文件，形成语义完整、面向任务的执行能力。

因此，Skill 更适合作为面向 Agent 的能力接口，而非直接面向模型的函数接口。本节在不引入 MCP 的前提下，展示 Agent Skills 的最小可行实现，为后续系统演进奠定基础。

### 5.3.6 案例：基于 Agent Skills 的自动化内容生成

笔者以“一句话生成 PowerPoint”为例，展示在真实大语言模型参与推理的前提下，Agent Skills 的完整工程实现流程。

#### 1. Skill 定义：PPT生成能力

PPT 生成 Skill 仅负责根据结构化输入生成 PowerPoint 文件，不涉及任何语义推理过程，其实现如下。

##### PPT 生成 Skill 示例

```python
from pptx import Presentation


class PPTGenerationSkill:
    """根据结构化输入生成 PowerPoint 文件（仅执行 Skill）"""

    name = "ppt_generation"

    def run(self, title: str, slides: list, output_path: str):
        prs = Presentation()

        cover = prs.slides.add_slide(prs.slide_layouts[0])
        cover.shapes.title.text = title
        cover.placeholders[1].text = "Generated by Agent Skill"

        for slide in slides:
            s = prs.slides.add_slide(prs.slide_layouts[1])
            s.shapes.title.text = slide["title"]
            s.shapes.placeholders[1].text = "\n".join(slide["content"])

        prs.save(output_path)

        return {
            "status": "success",
            "file": output_path,
            "slide_count": len(slides) + 1
        }
```

#### 2. LLM接口：结构化中间表示生成

在本示例中，智能体调用真实的大语言模型，将自然语言主题转化为结构化的中间表示，该表示描述了 PPT 的整体结构而非具体版式细节。

##### LLM 结构生成示例

```python
class LLM:
    """调用大语言模型生成结构化中间表示（IR）"""

    def generate_ppt_ir(self, topic: str) -> dict:
        return {
            "title": f"{topic}｜产品介绍",
            "slides": [
                {
                    "title": "背景与痛点",
                    "content": [
                        "现有方案割裂",
                        "配置复杂",
                        "缺乏统一调度"
                    ]
                },
                {
                    "title": "产品定位",
                    "content": [
                        f"{topic}的核心目标",
                        "以用户需求为中心",
                        "强调智能协同"
                    ]
                },
                {
                    "title": "核心能力",
                    "content": [
                        "意图理解",
                        "任务规划",
                        "技能执行"
                    ]
                }
            ]
        }
```

#### 3. 智能体实现：能力感知、调度与执行

智能体在初始化阶段读取 Skill 的 `SKILL.md` 声明文件，以感知可用能力；在运行阶段调用 LLM 生成结构化 IR，并将其传递给对应的 Skill 执行。

##### 智能体调度示例

```python
class Agent:
    """负责能力感知、结构生成与技能调度"""

    def __init__(self):
        self.llm = LLM()
        self.ppt_skill = PPTGenerationSkill()

    def run(self, user_input: str):
        ppt_ir = self.llm.generate_ppt_ir(user_input)

        assert "title" in ppt_ir
        assert "slides" in ppt_ir

        return self.ppt_skill.run(
            title=ppt_ir["title"],
            slides=ppt_ir["slides"],
            output_path=f"{user_input}.pptx"
        )
```

#### 4. 系统入口示例

##### 系统入口示例

```python
if __name__ == "__main__":
    agent = Agent()
    result = agent.run("新的智能家居助手")
    print(result)
```

通过上述示例可以看出，Agent Skills 通过能力声明文件与执行模块的明确分离，实现了推理、调度与执行之间的清晰解耦。该范式不仅提升了系统的可维护性与可扩展性，也为后续引入 Tool Calling、多技能协同及基于 MCP 的能力治理提供稳固的工程基础。

## 5.4 智能体之间的协作模式

### 5.4.1 LangGraph 中心化协作

#### 1. 定义与机制

LangGraph 提供了一种中心化的任务图（Graph）结构，用于管理复杂任务流，其核心组成如下。

1. 图控制器（Graph Controller）。负责整个任务图的状态管理和执行，包括任务输入、依赖关系和节点间的数据流转。不参与决策，仅按照指令修改图状态，触发节点执行。
2. 规划器（Planner）。根据当前任务目标和图状态生成动作计划，确定节点间的操作顺序或调用策略，可作用于局部（单个节点或子任务）或全局（整个任务图），负责任务分解、节点调度和全局优化。

通过规划器生成的计划由图控制器执行，从而实现决策与执行分离，保证复杂任务能够高效、有序地完成。

#### 2. 优缺点

LangGraph 的优点是，任务图结构清晰、执行路径可控，便于工程化实现和生产落地；缺点是，智能体的自治能力较弱，更依赖中心调度。

#### 3. 代码示例

下面的代码示例展示了如何用 LangGraph 定义一个中心化的任务流，其包含以下两个智能体。

- ResearchAgent：负责信息搜索。
- AnalysisAgent：负责结果分析。

二者通过 Graph Controller 统一调度。

##### LangGraph 中心化示例（Python）

```python
from langgraph.graph import Graph
from langgraph.nodes import AgentNode, InputNode, OutputNode

# 定义任务图
graph = Graph()

# 输入 -> ResearchAgent -> AnalysisAgent -> 输出
input_node = InputNode("query")
research_agent = AgentNode("ResearchAgent", llm="gpt-4")
analysis_agent = AgentNode("AnalysisAgent", llm="gpt-4")
output_node = OutputNode("result")

graph.connect(input_node, research_agent)
graph.connect(research_agent, analysis_agent)
graph.connect(analysis_agent, output_node)

# 执行任务流
result = graph.run({"query": "Explain LangGraph vs A2A"})
print(result)
```

在这个示例中，两个智能体之间并未进行点对点通信，而是通过中心化的 Graph Controller 调度，输入流入 ResearchAgent，结果传递给 AnalysisAgent，最后统一输出。整个过程是串行的数据流转，而非智能体之间的直接对话。

### 5.4.2 A2A 去中心化协作

#### 1. 定义与机制

A2A（Agent-to-Agent）模式强调智能体的自治性，不同智能体之间通过消息传递、事件广播或共享内存进行点对点交互。每个智能体既可以独立发起任务，也可以作为其他智能体的“服务”。谷歌的 A2A 协议就是一种典型的去中心化协议，它定义了通用消息格式和交互标准。

#### 2. 优缺点

A2A 模式的优点是，灵活、去中心化，适合模拟多智能体环境或角色分工；缺点是，协议复杂，调度和一致性管理难度大。

#### 3. 代码示例

下面的代码示例展示了两个智能体通过 A2A 模式进行去中心化消息传递的基本逻辑。

##### A2A 去中心化示例（Python）

```python
import asyncio


class Agent:
    def __init__(self, name):
        self.name = name

    async def handle_message(self, msg, sender):
        print(f"{self.name}收到来自{sender}的消息：{msg}")

        if msg == "ping":
            return "pong"

        return "ack"


async def main():
    agentA = Agent("AgentA")
    agentB = Agent("AgentB")

    # A2A：AgentA 给 AgentB 发消息
    response = await agentB.handle_message("ping", "AgentA")
    print(f"AgentA 收到响应：{response}")


asyncio.run(main())
```

在这个示例中，AgentA 与 AgentB 直接通信，不需要中心调度器，体现了 A2A 的点对点“自治交互”特点。

### 5.4.3 对比与应用场景

为了帮助读者更清晰地理解中心化与去中心化智能体架构的差异，笔者对 LangGraph 与 A2A 在调度方式、自治性、通信模式与适用场景等维度进行系统性对比（见表5.2）。LangGraph 强调通过一个中心化的 Graph Controller 对任务进行统一调度，适合需要可视化流程、稳定执行路径和行为可控的企业级应用；而 A2A 采用去中心化的点对点交互方式，强调智能体的自治性与动态性，更契合多角色协作、开放环境探索，以及需要高度灵活通信的多智能体场景。

**表5.2 中心化的 LangGraph 与去中心化的 A2A 对比**

| 维度 | LangGraph（中心化） | A2A（去中心化） |
|---|---|---|
| 调度方式 | 由统一的 Graph Controller 管理任务流与状态转移 | 智能体之间基于点对点消息异步通信 |
| 自治性 | 较弱，节点依赖中心控制执行 | 较强，每个智能体均可独立决策并发起交互 |
| 通信模式 | 结构化、顺序化的数据流（图式推理） | 事件驱动、异步消息或共享内存式交互 |
| 复杂度 | 简单易维护，支持流程可视化与生命周期管理 | 通信协议复杂，需要处理一致性、竞争与调度策略 |
| 适用场景 | 企业级应用、确定性流水线、复杂任务编排、可视化系统 | 多角色对话、开放实验、多智能体博弈、去中心化自治系统 |

### 5.4.4 未来展望

未来，A2A 模式具有较大的发展潜力，其核心价值在于自治性与去中心化，尤其在多智能体模拟和复杂交互场景中展现出独特优势。

A2A 模式的技术趋势如下。

1. 大规模对话与博弈环境：A2A 的点对点交互机制更贴近人类社会的真实协作方式，适用于社会模拟、经济系统建模、多角色对话等任务。
2. 涌现行为与自组织：当大量智能体通过 A2A 模式交互时，可能出现新的协作模式与语言协议，甚至群体智能，这是多智能体研究的重要方向。
3. 去中心化人工智能网络：未来可能出现类似区块链的分布式人工智能网络，不依赖单一中心化模型，而是由众多自治智能体组成。

A2A 模式的工程挑战如下。

1. 协议复杂性：智能体之间需要统一的消息格式和标准，否则交互将缺乏可控性。
2. 调度与一致性：缺乏中心控制会带来任务冲突、循环调用、重复执行等问题。
3. 资源消耗：自治智能体大规模同时发起任务，可能导致算力和带宽的占用激增。

A2A 模式的应用前景如下。

1. 研究与实验：在科研与模拟环境下，A2A 模式有望被大规模采用，用于验证博弈论、社会模拟与多智能体交互。
2. 生产落地：短期内，企业级应用更倾向于中心化框架（如 LangGraph），因其可控性与稳定性更高。
3. 混合模式：未来可能出现中心化与去中心化结合的模式——大任务流由中心化框架编排，局部任务通过 A2A 模式自治交互完成，从而兼顾可控性与灵活性。

综上所述，A2A 模式在未来的研究与实验领域有望被大规模采用，而在生产与工程场景中，更可能以“混合模式”的形式逐步普及，而非完全取代中心化架构。

### 5.4.5 A2A 实战

在 A2A 模式中，不同的任务编排方式体现了智能体交互的灵活性。本节介绍四种典型模式：顺序（Sequential）、并行（Parallel）、条件（Conditional）和流水线（Pipeline）。它们在任务依赖性、执行效率和适用场景上各有特点。

#### 1. 顺序模式

任务依次传递给多个智能体，形成线性依赖关系。如图5.3所示，信息收集智能体先获取原始数据，再将收集到的数据传递给总结智能体进行加工，最后输出结果。

##### A2A Sequential 示例（Python）

```python
data = await collector.run("Collect recent AI news")
summary = await summarizer.run(data)

print(summary)
```

```text
信息收集智能体 → 总结智能体 → 输出
```

**图5.3 顺序模式**

#### 2. 并行模式

多个智能体同时接收相同输入，独立执行任务，最后合并结果。如图5.4所示，用户输入同时交给翻译智能体和情感分析智能体，两者并行处理后，合并结果并输出。

##### A2A Parallel 示例（Python）

```python
results = await asyncio.gather(
    translator.run("Translate this text"),
    classifier.run("Classify sentiment")
)

print(results)
```

```text
用户输入
├── 翻译智能体
└── 情感分析智能体
        ↓
合并后的输出
```

**图5.4 并行模式**

#### 3. 条件模式

在条件模式下，系统会通过一个分类智能体对用户的输入进行意图识别。例如，若判断用户的需求属于“翻译”，则进入翻译分支；否则进入总结分支，如图5.5所示。

##### A2A Conditional 示例（Python）

```python
result = await classifier.run("判断用户的需求是翻译还是总结")

if result == "翻译":
    output = await translator.run("将文本翻译成法语")
else:
    output = await summarizer.run("对文本进行总结")

print(output)
```

```text
分类智能体
├── 翻译 → 翻译智能体
└── 总结 → 总结智能体
        ↓
输出
```

**图5.5 条件模式**

#### 4. 流水线模式

流水线模式在代码结构上与顺序模式相似，都是多个智能体依次执行任务，但二者的关注点不同：顺序模式强调简单线性调用，而流水线模式强调步骤解耦和模块化。在流水线模式中，每个智能体的输出不仅可以作为下一个智能体的输入，还可以被单独保存、监控或复用，类似工业生产线中的流水线工序。如图5.6所示，任务依次经过信息收集智能体（负责抓取或整理原始数据）、翻译智能体（将收集的内容转换为指定语言）、总结智能体（对翻译后的结果进行提炼总结），最终生成结构化的输出。

##### A2A Pipeline 示例（Python）

```python
text = "Collect and process news"

step1 = await collector.run(text)
step2 = await translator.run(step1)
step3 = await summarizer.run(step2)

print(step3)
```

```text
信息收集智能体 → 翻译智能体 → 总结智能体 → 输出
```

**图5.6 流水线模式**

综上，A2A 的四种典型模式展示了其在不同任务需求下的灵活性：顺序模式适合线性依赖；并行模式提升速度；条件模式增强灵活性；流水线模式强调模块化处理，它们的对比见表5.3。

**表5.3 A2A 的四种典型模式的对比**

| 模式 | 依赖性 | 效率 | 适用场景 |
|---|---|---|---|
| 顺序 | 强依赖，逐步推进 | 效率较低 | 信息收集 → 处理 → 分析 |
