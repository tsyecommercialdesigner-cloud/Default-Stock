# 引言

一张地图不仅要标出道路，也要标出边界。

所谓“任务不适合直接交给 Codex”，并不是说 Codex 完全不能参与，而是说：

> **该任务还没有被设计成可安全委派的工程任务**。

最常见的边界有以下4类。

# 目标模糊

例如“提升这部分能力”、“优化一下系统”，这类描述缺乏范围和验收标准，Codex 只能猜。

更合适的做法是：

> 先让 Codex 帮助澄清方案、列出候选改动和风险，再由人选择可执行的具体任务。

# 验证缺失

若：

> 关键逻辑既无测试，也无对照数据或可重复检查手段，却要求 Agent 直接大幅修改。

这并非高效，而是在放大不确定性。

若任务重要到值得委派，就应先补齐最小验证条件。

# 权限与回滚边界不清

像是：

- 生产写操作
- 敏感凭证
- 不可逆删除
- 受监管数据处理

都不应在**没有审批和回退路径**的情况下直接推进。

Codex 可以帮助准备脚本、Runbook、检查项和风险说明，但**关键执行点必须有明确的人工判断**。

# 跨系统复杂度尚未拆解

当一个任务同时牵涉架构、数据库、接口、前端和发布节奏，且需求仍然模糊时：

> 直接端到端委派通常只会叠加多个风险。

更稳妥的做法是：

> 先进行由开发者主导的拆解，再把局部可验证的子任务委派给 Agent 。

# 附录：高风险任务的受控交接示例

把高风险任务改写成受控任务，通常比直接禁止更实用。

下面这个 Markdown 示例把直接执行改成了准备变更说明、罗列风险并等待人工确认。

```markdown
# 任务目标
为生产配置改动准备变更说明，不直接执行写操作。

# 需要说明的条目
- 列出涉及的配置项
- 说明可能影响的服务
- 写出回滚步骤
- 给出执行前检查项
- 标出必须等待人工确认的节点
```

它并没有把高风险任务完全排除在 Codex 之外，而是将其压缩到一个更适合 Agent 参与的阶段：

> 先分析、准备、暴露风险，再等待受控执行。

因此，边界并非“哪些事永远不能做”，而是“哪些事不能在任务设计不合格时直接推进”。

只要**目标**、**验证**、**权限**、**审批**和**回滚**仍说不清楚，换任何入口都不可能让任务自动变得安全。
