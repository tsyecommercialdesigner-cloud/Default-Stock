# 第 2 章 AI 编程的最小原理——知道边界在哪里

Codex 最容易让人误判的地方，不是它写不出代码，而是它太像一个已经透彻理解了问题的工程师。它能根据需求进入代码库，找到相关文件，补齐实现，运行测试，再根据报错继续修改。很多时候，结果来得又快又完整，让人很自然地产生一种错觉：既然 Codex 已经能把任务做成这样，剩下的判断似乎也可以一并交给它来完成。

AI 编程真正的分水岭，恰恰在这里。会生成代码，不等于看清了整个项目；能修复报错，不等于理解了所有业务约束；局部修改看起来正确，也不等于工程决策就是合适的。想要用好 Codex，工程师首先要知道它为什么能写代码，它完成任务时依赖什么信息，又会在哪些边界上犯错。

先看一个足够小的失败案例。你让 Codex 给用户资料接口增加 `nickname` 字段，它根据模型文件和接口返回值很快补齐了实现，相关测试也通过了。但上线前你才发现，移动端使用的是另一套序列化字段，后台导出任务也依赖旧字段列表。问题不在于那几行代码写错了，而在于任务一开始就没有把“字段会穿过哪些边界”说清楚。

本章讨论的边界，主要就是为了避免这种“局部正确、整体遗漏”的失败。读者不需要退回到完全人工开发，而是要学会在任务开始前暴露影响面，在执行过程中观察证据，在结果出来后按风险进行验证。

## 2.1 大语言模型为什么能处理代码

代码和自然语言很不一样。自然语言允许含糊、跳跃和省略，一句话即使语法不完整，人也常常能猜出意思；代码则严格得多，少一个括号、错一个缩进、传错一个参数，都可能导致程序无法运行。也正因为如此，很多工程师第一次看到大语言模型写代码时，会本能地觉得奇怪：一个靠生成文本工作的模型，为什么能处理如此精确的东西？

答案恰恰藏在代码的这种“精确”里。

对大语言模型来说，代码首先是一种由符号组成的序列。函数名、关键字、变量、括号、运算符、缩进关系，都会成为它判断后续内容的线索。大语言模型并不是像编译器那样先建立一个完整的语法树，再像工程师那样逐步设计实现，而是在已有上下文中，根据训练中学到的大量结构和模式，生成接下来更可能成立的内容。

代码比自然语言更适合这种生成机制，原因在于其约束更强，合法写法的空间也随之收窄。看到一个函数定义，接下来通常是函数体；看到一次数据库查询，后面往往伴随结果处理、异常处理或返回值封装；看到测试文件中既有的写法，新的测试大概率会沿用相同的 fixture、断言风格和命名方式。代码的结构性，让大语言模型不必在无限开放的表达空间中寻找答案，而是在一组更明确的候选路径中继续往下写。

更重要的是，工程代码并不是一堆孤立的符号。人类为了让代码可维护，会不断在代码中留下意图：`calculate_order_total` 暗示要处理订单金额，`validate_email` 暗示要做合法性校验，`is_authenticated` 暗示返回的是一个判断结果。目录结构、类型注解、测试名称、接口字段，都会把“这段代码要做什么”暴露出来。良好的工程实践本来就是为了辅助人类协作，到了 AI 编程时代，它们也在帮助大语言模型更准确地获取上下文。

这就是为什么同样一句“帮我补一下逻辑”，把它放进不同代码库里，Codex 的表现差异会很大。在命名清晰、模块边界明确、已有实现可参考的项目中，它很容易按照现有模式继续工作；而在职责混乱、文件庞大、约定全靠口头传递的项目中，它就更容易误读意图，写出局部像样但整体不合适的代码。

除了结构和语义，代码还有一个优势：大量日常工程任务本身就具有很强的重复性。新增接口、补表单校验、处理分页查询、封装错误响应、补日志、写回归测试，这些工作并不“简单”，但它们确实在不同项目中反复出现。大语言模型在大规模代码和技术文本中见过大量相似模式，因此在这些高频任务上，往往能很快生成一版看起来相当成熟的代码实现。

这里的关键不是把 Codex 理解成一台“背过很多代码答案”的机器，也不是把它理解成一位已经拥有完整工程经验的程序员。更准确的说法是：它能利用训练中习得的模式，结合当下的代码上下文，快速生成一个很像工程解法的候选结果。

这解释了 AI 编程中最重要的一条经验：对于越接近通用工程模式的任务，Codex 越容易发挥能力；对于越依赖具体业务规则、历史包袱和隐性约束的任务，Codex 越需要人类工程师把上下文讲清楚。参考现有接口新增一个相邻功能，与独立制定一套复杂促销系统的定价规则，在难度上并不处于同一等级。

因此，理解 Codex 为什么能写代码，并不是为了神化它，而是为了把它放回正确的位置。它之所以有用，是因为代码本身足够结构化，工程上下文中充满语义线索，而软件开发又存在大量可复用模式。它的能力从这里“长”出来，它的边界也从这里开始显现。

## 2.2 Codex 完成任务时依赖什么

在实际工程中，Codex 之所以比一段孤立的聊天回复更有用，是因为它能围绕任务建立上下文，读取项目文件，执行命令，观察反馈，再继续修改。

换句话说，AI 编程不是一次回答，而是一段工作过程。

### 2.2.1 上下文决定 Codex 此刻能看到什么

大语言模型每次处理任务，都受限于上下文窗口的容量。你可以把容量理解为大语言模型在当前推理中能同时参考的信息上限。

这个窗口通常需要容纳：

- 你的任务描述；
- 项目中的相关代码和配置；
- 约束条件和已有文档；
- 工具调用结果；
- 测试日志、报错信息和执行历史；
- 当前会话中已经形成的结论。

即使上下文窗口再大，也无法让整个代码库都在“眼前”。一个项目往往包含大量模块、测试、配置、脚本、依赖说明和历史约定，Codex 必须在有限空间内筛选出当前最相关的信息。

这直接导致 Codex 不是基于“对整个项目的全知视角”工作，而是基于“当前可见上下文”工作。上下文足够聚焦时，它会表现得很懂项目。上下文缺失、噪声过多或关键约束超出窗口覆盖范围时，它就可能写出局部合理却全局不合适的代码。

### 2.2.2 文件读取和检索帮助 Codex 补足项目背景

代码库通常远超单次上下文窗口的承载上限，AI 编程工具因此需要不断“找资料”。它可能通过文件名、目录结构、关键词、符号引用、已有测试和调用关系，定位与当前任务有关的内容。你可以把这理解成一种面向项目的查阅能力：先找可能相关的材料，再把关键片段放进当前工作上下文中。

这也是为什么下面这些工程习惯会直接影响 AI 协作质量：

- `user_auth_service.py` 比 `service.py` 更容易定位；
- `src/payment/` 比一堆堆在 `utils/` 中的支付逻辑更容易理解；
- 清楚的测试名比含糊的测试名更能表达预期；
- 模块文档、接口注释和迁移说明会补充代码本身没有明示的约束。

检索和文件读取能扩展视野，但不能保证全知。Codex 仍然可能找错文件、遗漏隐式依赖，或者把两个名称相近的模块混在一起。

对于具备代码检索能力的编码 Agent，其处理超大代码库的方式可类比于 RAG（见图 2-1）：先按任务检索相关上下文，再基于这些上下文进行分析和生成。Codex 的实际工作流还可能结合文件搜索、代码读取、工具调用和多轮验证，因此不一定等同于单一的标准 RAG 流程。

这种机制也可以类比为“翻书”：上下文窗口像当前摊在桌面上的工作材料，容量有限，但随时可用；检索和工具调用则好似去代码库或外部资料中查阅相关内容，再把有用的信息带回当前工作过程。上下文窗口可视作“工作记忆”，容量有限但随时可用。RAG 则可视作“查阅资料”能力，可以在更大范围内搜索信息，但每次只能取回一小部分放入工作记忆。

检索过程通常依赖以下两个信号：

- **关键词匹配**：根据任务描述中的关键词（如“用户”“认证”“JWT”），在代码库中搜索包含相同关键词的文件和代码片段。
- **语义相似度**：使用向量嵌入（embedding）技术，将代码片段和任务描述都转换为数学向量，然后计算向量间的距离以衡量语义相关性。

### 2.2.3 执行反馈让生成进入工程闭环

大语言模型能够生成代码，但软件工程并不以“生成一段代码”为终点。代码必须在项目环境中，接受语法、依赖、类型、运行时行为和既有测试的检验。因此，编码 Agent 与单纯的代码补全拉开了距离：编码 Agent 不仅给出一个看似合理的实现，而且可以把实现放回项目中，观察其是否成立。

这种执行反馈机制改变了 AI 编程的性质。

如果没有执行反馈，大语言模型给出的代码本质上仍是一种基于上下文的推断。它可能符合常见写法，也可能贴近当前项目风格，但这些都还停留在“看起来可能正确”的层面。只有当代码经过测试、构建、静态检查，或者至少在最小场景中实际运行之后，生成结果才开始与项目事实发生碰撞。

这种碰撞很重要。一个导入路径是否存在，一个参数签名是否匹配，一个测试假设是否被违背，一个修改是否触发了更深层的兼容性问题，往往不是单靠局部阅读就能完全确认的。执行反馈把这些隐藏在工程环境中的约束重新暴露出来，使 Codex 能够根据事实修正先前的推断。

因此，AI 编程中的“反馈”并不是生成之后附带的一道检查工序，而是任务完成机制的一部分。它把一次性输出变成多轮逼近：先根据上下文形成候选实现，再通过项目自身的反馈排除错误，随后继续调整。大语言模型负责提出和修正候选方案，而项目本身的验证流程则通过测试、构建和运行，不断回答一个更根本的问题：这段修改在这里究竟是否成立。

从这个意义上说，测试体系、类型约束、lint 规则、构建脚本和可复现的报错信息，不仅仅是传统开发流程中的质量保障手段，也是 AI 编程协作赖以成立的外部校验机制。项目越能清楚地暴露错误，Codex 就越不容易把“看似正确”误当成“已经完成”。

### 2.2.4 能力再强，也不能替代正确的上下文

执行反馈能够纠正一部分错误，却不能自动弥补所有信息缺口。因为反馈检验的是已进入工程过程的产物，而许多更关键的偏差，早在“应该看什么、应该改哪里、应该遵守什么约束”这一层面就已经发生。

这正是解决上下文问题的关键。

当 Codex 面对一个任务时，它并非从项目全貌出发作判断，而是从当前获得的信息出发作判断。它所看到的文件、检索到的片段、读到的测试、用户给出的描述，以及此前对话中保留下来的结论，共同构成了它此刻理解任务的基础。如果这个基础偏了，那么后续推理越流畅，错误就越不容易被察觉。

例如，一个项目同时保留两套接口实现，旧版本仍在仓库中，新版本才是当前入口。若任务描述和检索线索都把注意力引向旧实现，Codex 完全可能在错误的位置做出一组风格一致、测试局部通过的修改。又如，某项约束并不体现在代码表面，而来自团队长期形成的约定：不能新增依赖，必须兼容历史字段，某模块只能通过既有扩展点接入。若没有将这些约束纳入当前上下文，模型就只能参照可见材料和通用经验进行推断。

这里需要警惕的，并不是“模型能力不够强”这一空泛的判断，而是强能力在不完整上下文中会制造出一种更具迷惑性的完成感。它能够迅速找到一种解释，补出一套实现，并把局部细节处理得当。工程问题真正的难点，往往不在于把某段代码写出来，而在于确认这段代码是否写在正确的地方、是否服从正确的约束，以及是否回应了真正的问题。

因此，评价 AI 编程协作时，不能只看它写出了什么，更要追问它依据什么写出这些内容，它读到了哪些材料，哪些事实只是推断，哪些边界仍未被验证。只有把这些问题纳入协作过程，模型的生成能力才能被置于可靠的工程语境中，而不是在局部合理性中越走越远。

## 2.3 第一条边界：信息永远不完整

AI 编程的首要制约因素并不是代码生成能力，而是它所依据的信息根本不可能完备。

软件项目并不是一份可以一次读完就彻底理解的说明书。它由代码、配置、测试、依赖、文档、提交历史、团队约定和业务背景共同构成。其中有些信息写在显眼的位置，有些则分散在不同模块之间，还有一些只存在于长期演化形成的惯例之中。即使是经验丰富的工程师，接手陌生系统时也需要不断查阅、验证和追问，才能逐渐形成对项目的可靠认知。

Codex 面对的也是同一种问题，只是这种问题更容易被它忽视。它能够迅速读取相关文件，沿着线索补齐实现，并把局部代码写得相当完整。于是工程师很容易把这种局部完成度误认为整体理解已经建立。实际上，它的每一次分析和修改，都只能基于当前进入任务过程的那部分信息。那些没有被检索到的代码、没有被说明的约束，以及没有被测试覆盖的风险，并不会因为生成结果流畅而自动消失。

因此，AI 编程的第一条边界可以表述得朴素一些：它永远是在不完整的信息中工作。真正的问题不在于能否彻底消除这种不完整性，而在于能否识别哪些缺口会改变判断，哪些风险需要通过检索、说明和验证尽早暴露出来。

一个实用的判断办法是把任务转化成影响面问题：它会不会改变数据结构、接口协议、权限判断、缓存策略、异步任务、前端展示或外部调用方？如果答案涉及两项以上，就不应只让 Codex“直接实现”，而应先让它列出可能受影响的文件、测试和未知点，再决定是否进入修改阶段。

### 2.3.1 项目全貌不会自动进入当前任务

真实项目的复杂性往往不在于某段代码有多难写，而在于许多局部共同构成了一个高度耦合的整体。一个字段的变化，可能同时影响数据库模型、接口协议、序列化逻辑、前端展示、迁移脚本和回归测试；一个看似独立的功能开关，背后也可能牵涉权限、监控、灰度发布和兼容策略。

但 Codex 开始执行任务时，并不会自动掌握这个全貌。它通常从任务描述出发，读取最相关的文件，然后沿着搜索结果和调用关系逐步扩大视野。对于边界清晰、影响面较小的修改，这种方式往往已经足够。例如，为一个已有函数补充测试时，参考现有写法即可，真正需要纳入上下文的材料也许只有实现代码、测试用例和少量配置。

一旦修改跨越多个层次，情况就会有所不同。若任务同时牵涉数据结构、接口行为和外部调用方，而上下文仅覆盖其中一部分，就很容易把“已经改到的地方”误认为“所有需要改的地方”。这类问题并不总会表现为明显报错。很多时候，局部实现可以顺利通过测试，而遗漏的问题却留在尚未触及的模块中，直到集成、上线或后续维护时才显现出来。

因此，面对复杂任务时，不能只要求 Codex“完成功能”，还要追问这次变更将涉及哪些边界。若没有看清影响面，则完成不过是一种过早到来的错觉。

### 2.3.2 最容易缺失的是代码没有明示的关系

并不是所有工程关系都能通过一次显式引用被发现。诸如代码清单 2-1 所示的依赖，通常比较容易追踪。

**代码清单 2-1：可通过显式导入追踪的代码依赖**

```python
from serializers.order import serialize_order
```

导入关系、类型签名、函数调用和模块引用，会给开发者和 AI 编程工具留下清晰线索。沿着这些线索搜索，往往能够界定修改所涉及的范围。

但项目中还有另一类关系，它们并不总以直接调用的形式出现（见代码清单 2-2）。注册表、配置驱动、反射、事件订阅、字符串协议、插件机制、跨服务契约等，都会把实际的关联藏到代码表面之下。

**代码清单 2-2：隐藏在注册表中的隐式调用关系**

```python
order_data = serializer_registry["order"](order)
```

这里的序列化行为依赖注册关系，而非一眼可见的函数调用。若 `order` 字段发生变化，则需要同步检查的范围也许并不局限于当前文件，而是延伸至注册逻辑、协议定义，甚至另一个服务的消费端。

这类隐式关系之所以危险，是因为不仅难以搜索，而且其往往承载着系统演化过程中所积累的隐性约定。代码仍然能运行，并不意味着契约没有被破坏；测试没有立刻失败，也不意味着外部依赖没有受到影响。

因此，系统越依赖动态机制、弱类型边界和隐含协议，AI 编程协作就越不能只依赖“读到的代码”。关键约束需要用文档说明，关键路径需要被测试覆盖，任务描述也需要明确指出那些仅靠局部阅读不容易发现的关联。

### 2.3.3 信息会在长任务中逐渐变形

上下文信息的不完备并非仅存在于任务启动阶段。随着任务持续推进，先前已经获得的信息也可能失效或过时。

在一次持续时间较久的协作中，任务描述、代码片段、搜索结果、测试日志、失败尝试和中间结论会不断累积。它们共同构成当前工作过程的背景，却不具有同等价值：有些是稳定约束，有些只是某次尝试留下的临时判断，还有些结论早已被推翻，却仍可能在后续对话中占据注意力。

这会带来一种典型的偏移：任务最初关注的是一个明确的问题，但随着修复过程的推进，注意力逐渐被报错信息、局部实现和补丁细节吸引。等到代码终于“看起来能跑”，最初的目标、边界和验收标准反而变得模糊了。

对 AI 编程来说，这种偏移尤其需要警惕。因为 Codex 能够持续接住局部问题：一个报错接一个报错地修复，一个文件接一个文件地修改。局部推进越顺畅，就越容易掩盖一个更根本的问题：当前正在解决的，是否仍然是最初那个问题。

因此，长任务需要主动收束上下文范围。工程师应在关键节点重新确认目标、约束和剩余风险，把已经稳定的结论从临时尝试中分离出来；当任务已经扩展至多个相对独立的问题时，也应及时拆分，而不是让所有信息继续堆积在同一条工作线上。

### 2.3.4 项目结构决定上下文的质量

很多人一提到 Harness 工程，首先想到的是如何写提示词。提示词当然重要，但对编程任务而言，更深层的 Harness 工程往往体现在代码库本身。

一个结构清晰的项目，会不断向协作者提供判断线索。目录能够说明职责边界，文件名能够提示业务语义，类型和接口能够约束使用方式，测试能够表达行为预期，文档能够补足代码没有写出的背景。人类工程师依赖这些线索建立理解，Codex 同样如此。

反之，如果一个项目把大量业务逻辑堆积到语义模糊的 `utils`、`helpers` 和 `misc` 中，关键规则只靠口头传递，测试只覆盖表面路径，接口兼容性没有任何记录，那么 AI 并不是唯一会感到困惑的协作者。只是它可能更快地在这种混乱中生成一版“看似合理”的实现，从而把结构性问题暂时遮住。

因此，提升 AI 编程协作质量的关键，并不在于为任务撰写一段更长的说明，而在于从根本上让代码库本身更容易被理解：

- 让模块职责尽量聚焦，避免单个文件承载过多无关逻辑；
- 让目录和命名表达业务含义，而不只是承担存放功能；
- 让关键接口说明输入、输出、依赖与兼容性边界；
- 让测试不仅验证结果，也表达重要行为预期；
- 让团队约定和历史决策有可查的记录。

这些做法原本就是良好的工程实践。AI 编程并没有削弱它们的重要性，反而让它们的价值更加直接：项目越能把自身的结构、约束和意图显露出来，Codex 就越有可能依据有限的信息作出接近正确的判断。

## 2.4 第二条边界：Codex 会犯三类工程错误

在实际编程中，AI 编程常见的失败模式可归纳为三类：幻觉、漏改和误判。它们不是同一种错误，检测方式也不同。

### 2.4.1 幻觉：不知道怎么写，却写得很像

如果你用过 AI 编程工具，那么一定见过这样的场景：让它根据一个陌生框架写一段代码，结果运行后发现根本跑不通；或者它只修改了一个文件，却忘了修改另一个依赖该文件的文件；或者它把简单的 bug 修复搞成了大规模重构。这些都是 AI 编程中常见的代码幻觉（见图 2-2）。

> **图 2-2** AI 编程中常见的代码幻觉（原 PDF 中为分类示意图）

幻觉最容易被察觉，因为它常常表现得很直接，如：

- 调用不存在的 API（应用程序编程接口）；
- 混用不同语言或版本的语法；
- 写出结构合理但事实不成立的参数、配置项或返回值处理逻辑。

幻觉还有一种更隐蔽的形式：代码在特定版本中可以运行，但在项目当前锁定的版本中不能运行。模型“记得”一种常见写法，不等于知道你当前环境的实际依赖版本。

应对幻觉的核心是让事实尽快接受检查：

- 让代码运行起来；
- 运行测试、构建、lint 和类型检查；
- 对陌生库和关键 API 核查项目实际版本与官方用法；
- 审查新增的 import、依赖和配置项；
- 对小众库、内部 SDK（软件开发工具包）和版本敏感接口，给出文档或现有示例。

幻觉不是靠“相信模型会更认真一些”就能解决的，而应依靠验证机制将其暴露出来。

### 2.4.2 漏改：局部正确，整体不完整

漏改比幻觉更常见，也更贴近实际工程问题。假设你让 Codex 把 `username` 拆分成 `first_name` 和 `last_name`。它可能改了模型和接口，却忘了改序列化器、迁移脚本、前端展示、测试用例和相关文档。

一次典型漏改如代码清单 2-3 所示。

**代码清单 2-3：漏改示例**

```text
已修改：
src/models/user.py
src/api/users.py

可能遗漏：
src/serializers/user_serializer.py
migrations/
tests/
docs/api/users.md
```

漏改的典型特征是，每个已被修改的文件都可能看起来没问题，但改动集合不完整。应对漏改的关键举措是主动扩大“需要检查的范围”：

- 在任务开始时明确变更面，尤其是跨模型、接口和数据层的改动；
- 搜索待修改字段、函数名、接口路径和配置键的引用；
- 检查测试、迁移、文档、序列化和调用方是否同步更新；
- 用 `git diff --stat` 和完整 diff 检查变更范围是否符合预期；
- 运行相关模块的测试，必要时执行更大范围的回归验证。

测试能发现很多漏改，但那些还没写成测试用例的风险、还没被识别的风险，测试是发现不了的。没有测试或测试薄弱的地方，仍需依赖人类审查和结构化搜索。

### 2.4.3 误判：代码没错，决策不对

误判最值得警惕，因为它往往不会立刻报错。例如，项目原本统一使用标准库缓存方案，即使你仅要求“给这个函数加缓存”，Codex 也可能引入一个新的依赖，如代码清单 2-4 所示。

**代码清单 2-4：引入依赖示例**

```python
from cachetools import TTLCache, cached

@cached(cache=TTLCache(maxsize=100, ttl=300))
def get_user(user_id: int) -> User:
    ...
```

这段代码可能语法正确，行为也满足局部需求，但却带来了新的工程问题：

- 增加外部依赖；
- 偏离项目已有缓存策略；
- 增加维护和安全评估成本；
- 导致同类实现出现不一致。

误判不是源于“不会写代码”，而是源于缺少项目的隐性背景：团队约定、历史决策、兼容性承诺、业务优先级和维护成本。

应对误判的关键在于人类工程师的深度参与：

- 把关键约束显性化，例如“不要新增依赖”“保持 API 向后兼容”“沿用现有缓存方案”；
- 在项目文档中记录技术栈约束、风格约定和关键决策；
- 审查依赖变更、接口调整和架构层改动；
- 区分实现问题与决策问题。函数实现可以交由验证闭环检验，方案该不该这样设计仍需人工把关。

对于幻觉、漏改和误判，暴露问题的方式并不相同：幻觉往往要靠运行和测试尽快揭穿；漏改需要靠搜索引用、检查变更面和回归验证来发现；误判的认定则更依赖人类工程师对约束、依赖和架构决策的审查。只有理解了这一差异，2.5 节的协作方法才有针对性。

## 2.5 从边界意识到协作方法

理解这些边界之后，AI 编程应被理解为一种受约束的工程协作：工程师提出目标，提供必要背景，保留关键判断，再用验证结果检验修改的合理性。Codex 能加快其中许多环节，但不能替代这些环节。

这解释了为什么同一个任务，换一种交代方式，其结果会出现明显差异。“优化一下用户模块”几乎没有给出可执行边界；“为注册接口补充邮箱重复校验，保持现有响应格式，并补充回归测试”则同时说明了目标、约束和验收方向。前者把大量判断留给模型猜测，后者把任务放回工程语境中。AI 编程协作的关键不在于把提示词写得花哨，而在于尽量减少那些本不该由模型猜测的部分。

上下文同样如此。API、内部协议、特殊业务规则、历史兼容性要求，这些信息如果不进入任务过程，Codex 就只能依据眼前代码和既有模式作出推断。推断结果有时会很接近答案，有时却会把局部经验误当成项目事实。人类工程师真正需要补充的，不是越多越好的材料，而是能改变推断结果的材料：相关实现、关键文档、错误日志、数据样例，以及当前变更不能突破的边界。

对于有跨文件修改的任务，边界意识还意味着先确认影响面，再追求生成速度。字段重命名、接口调整、数据结构变化，看起来只是修改一个定义，实际上却牵动调用方、序列化层、迁移脚本、测试和文档。让 Codex 先沿着引用关系和现有实现查清范围，再进行修改，通常比直接要求“完成改造”更接近实际工程流程。这样做并非降低效率，而是在更早的阶段避免漏改。

最后，验证结果必须回归到协作的中心环节，即由人类工程师作出最终判断。代码生成得再流畅，也不能把“看起来完成了”当作“已经成立”。运行、测试、构建、静态检查和 diff 审阅，分别在不同层面暴露问题：有些揭穿虚假的 API 和语法错误，有些发现关联修改没有同步，有些提示人类工程师这次变更已经越过了原本的设计边界。在依赖引入、兼容性、安全和业务规则等问题上，仍然由人类工程师作出最终判断。

因此，本章讨论边界并不是为了给 AI 编程泼冷水。只有知道 Codex 依赖什么、会在哪些地方失真，人类工程师才有可能把它纳入稳定的交付流程。

## 本章小结

Codex 之所以能写代码，是因为代码高度结构化、自带大量语义线索，且工程任务中存在大量可复用模式。它能基于上下文生成很强的初稿，并借助文件读取、命令执行和验证反馈推进任务。

但 Codex 并不是一个能看全整个项目、什么都懂的工程师。它的能力取决于当前可见的上下文、项目结构、文档、任务约束和验证闭环。信息不完整时，它会出现三类典型错误：

- **幻觉**：不知道怎么写，却写得很像。
- **漏改**：局部正确，整体不完整。
- **误判**：代码没错，决策不对。

人类工程师要做的，不是盲信，也不是拒绝，而是把这些边界纳入协作过程：让目标更明确，让上下文更可靠，让验证更及时，并在关键工程判断上始终承担最终责任。

理解这些之后，下一步不是继续停留在“AI 到底会不会写代码”的争论中，而是把上下文、约束和验证组织成一个可复用的工作流。
