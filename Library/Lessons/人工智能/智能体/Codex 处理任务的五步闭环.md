# 概览

每次让 Codex 处理代码任务，都可以按以下 5 步进行：

1. 【需求澄清】：先确认自己到底想改什么。
2. 【任务下达】：把目标、范围、约束和验收标准说清楚。
3. 【执行观察】：关注 Codex 读了什么、准备改什么、如何验证。
4. 【结果验证】：审查 diff，运行测试，确认没有意外改动。
5. 【代码提交】：确认结果可接受后，再把变更纳入版本历史。

这 5 步并不繁琐。相反，它们是你把 Codex 从“代码生成器”变成“工程协作工具”的基础。

Codex 可以加快其中许多步骤，但这些步骤不能被省略，即使是一次很小范围的修改，也要：

1. 先限定范围
2. 再理解上下文
3. 然后实施变更
4. 最后用验证结果校正判断

哪怕只是修改一个格式化函数，也应该知道改了哪里、测试了什么，以及是否涉及不相关模块。
更复杂的任务本质上仍然遵循这个闭环，变化的只是每一步的复杂度，而不是流程本身。

---
# 第一步：需求澄清

所谓需求澄清，并非指描述越长越好，而是要把那些**影响实现判断的关键信息**交代完整。
很多不必要的重构，并不是因为任务复杂， 而是因为在没有理解上下文之前就急于动手。

需要明确：

- 目标是什么？
- 修改哪里？
- 新增行为的具体表现如何？
- 最终依据什么判断任务完成？

需求澄清的意义，在于让 Codex 开始执行前先划定任务的边界：

- 哪些变化是本次任务的目标？
- 哪些变化即使看起来合理，也不应在这一次修改中发生？

边界描述得越清楚，Codex 越容易把注意力放在真正需要改变的地方。

---
# 第二步：任务下达

在作为指令的提示词中，应包含以下 4 类关键信息：

- 变更目标
- 技术约束
- 文件范围
- 验收标准

这比一句“帮我优化一下”更适合工程协作，Codex 不需要猜范围，也不需要猜你是否接受重构。

---
# 第三步：执行观察

在 Codex 的执行过程中，会实时地返回它的进展情况，此时我们真正需要关注的不是动画和措辞，而是以下 3 个问题：

- 它是否读了相关文件？
- 它是否理解了修改边界？
- 它是否真的做了验证？

执行过程不可能每次都顺利，在遇到偏差时，需要及时修改上下文和边界：

- 如果 Codex 准备修改不相关的文件，那你就立即通过引导提示词纠正 Codex 的这种行为；
- 如果 Codex 尝试解决问题时一直没有抓住关键点，那你就直接指出问题出现的根本原因；
- 如果 Codex 已经做出了非必要的修改，可以只恢复该部分文件，并把边界重新告诉 Codex。

这些中断并不罕见。工程协作本来就包含澄清、纠偏和验证。
Codex 能加速执行，但**不能替你决定什么改动应该进入项目**。

---
# 第四步：结果验证

这一步是由人类工程师作出最终判断，此时绝不能把“看起来完成了”当作“已经成立”，工程质量绝不能建立在“看起来大概率没问题”的假设之上。

一个需求即使很小，也可能在实现过程中偏离原意，一次变更所带来的风险可能在不知不觉中超出了需求范围，而问题在于未必会出现明显的错误提示，有时只是一个不起眼的兼容性变更，却会在后续调用链中被逐级放大，最终演变为系统故障。在**依赖引入**、**兼容性**、**安全**和**业务规则**等问题上，必须由人类工程师作出最终的判断。

运行、测试、构建、静态检查和 diff 审阅，分别可以在不同层面暴露问题：

- 有些揭穿虚假的 API 和语法错误；
- 有些发现关联修改没有同步；
- 有些提醒人类工程师这次变更已经越过了原本的设计边界。

测试同样需要被审视。新增测试并不等价于新增了有效保障。一个测试可能只验证了实现写法，却没有验证业务真正关心的输出；也可能只覆盖了新分支，却没有确认旧行为仍然有效。对维护者而言，重要的不是“Codex 写了测试”，而是：

> **整个测试是否真正覆盖了需求**。

测试通过固然重要。但测试失败时暴露的信息同样重要，其可能指出实现错误，也可能揭示环境配置、依赖状态或代码库中原本就存在的阻塞，关键在于：

> **辨别失败发生在哪一层**。

只有把实现问题、测试问题和环境问题区分开，后续处理才不会失焦。

验证还有另一层价值：

> 它让开发者保持对代码修改的决策权。

阅读 diff、检查改动范围、运行合适的测试，本质上是在回答几个问题：

- 这次修改是否只做了该做的事？
- 是否符合项目原有风格？
- 是否引入了新的复杂度？
- 是否真的经过了可重复的检验？

这些问题如果在合并前不作回答，往往就会在排查问题时被迫回答，而且代价更高。

因此，验证不是对 Codex 的不信任，而是软件工程中责任链条的一部分。Codex 可以帮助你更快地定位代码、生成补丁、补充测试，却不能替你承担代码进入仓库之后的后果。只要这段代码还需要继续维护，就应该在合并前被看懂、被检查、被验证。

---
# 第五步：代码提交

整理结果，决定是否提交。

此时，不应因为“改动很小”就直接提交。要先把以下这些事项写清楚：

- 差异
- 已跑验证
- 未跑通原因
- 剩余风险

当结果可被审查和接手时，再决定是否进入提交或后续修复流程。
