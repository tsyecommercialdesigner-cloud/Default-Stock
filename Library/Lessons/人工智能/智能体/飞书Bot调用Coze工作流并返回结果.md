# 🧩 整体架构（你看到的效果背后的真实流程）

## **1. 用户把视频上传到飞书文档 / 飞书聊天窗口**

飞书会生成一个 **文件对象（file_token）**。

## **2. 飞书机器人监听到文件上传事件**

机器人可以是：

- Coze 发布的飞书 Bot
    
- 自己写的飞书机器人（Node.js / Python）
    
- 飞书自动化（Webhook）
    

## **3. 机器人把 file_token 发送给 Coze 工作流**

Coze 工作流收到后：

- 下载视频
    
- 调用视频理解模型（如 Coze 内置的 Video QA）
    
- 执行你编排的步骤（拆解、总结、提取关键帧、生成文案等）
    

## **4. Coze 把处理结果返回给飞书机器人**

机器人再把结果：

- 发回飞书文档
    
- 或发回飞书聊天窗口
    
- 或写入飞书知识库
    

---

# 🧩 为什么看起来像“飞书文档自己在处理视频”？

因为：  
**飞书文档只是展示层，真正的智能处理在 Coze。**

你看到的效果类似：

1. 把视频拖进飞书文档
    
2. 几秒后文档里出现“视频拆解结果”
    

但实际上：

- 飞书文档 → 触发机器人事件
    
- 机器人 → 调用 Coze 工作流
    
- Coze → 返回结果
    
- 飞书文档 → 展示结果
    

所以你问的这句话：

> “这是把飞书文档作为 Coze 交互层的前端显示吗？”

**答案：是的，本质上就是这样。**

飞书文档 = UI  
Coze = AI 引擎  
飞书机器人 = API 桥接层

# 🧩 你可以用 Coze + 飞书实现的三种方式

## ① **Coze 官方飞书 Bot（最简单）**

- 在 Coze 发布 Bot 到飞书
    
- 飞书里上传视频
    
- Bot 自动触发工作流
    
- 结果自动回到飞书
    

适合你：  
**不写代码，快速上线。**

---

## ② **飞书自动化（Webhook）+ Coze 工作流**

- 飞书自动化监听“文件上传”事件
    
- 调用 Coze 工作流 API
    
- 把结果写回飞书文档
    

适合你：  
**需要更灵活的触发条件。**

---

## ③ **自建飞书机器人（Node.js / Python）+ Coze API**

- 完全自定义
    
- 可做复杂逻辑
    
- 可接入你电商的内部系统
    

适合你：  
**你要做电商自动化平台级系统。**

# 🧩 你看到的“视频拆解”一般是这样实现的

### Coze 工作流步骤：

1. **下载飞书视频文件**
    
2. **调用视频理解模型（Video Understanding）**
    
3. **按你的需求拆解：**
    
    - 场景分段
        
    - 关键帧提取
        
    - 文案生成
        
    - 话术提取
        
    - 商品卖点分析
        
4. **生成 Markdown 文档**
    
5. **通过飞书 API 写回飞书文档**
    

所以你看到的“自动拆解视频”其实是：

**飞书文档只是展示结果，Coze 才是大脑。**

# 🧩 给你一个真实的链路示例（你可以直接复刻）

### 用户操作

把视频拖进飞书文档

### 背后发生的事

1. 飞书触发事件：`file.uploaded`
    
2. 飞书机器人收到 file_token
    
3. 机器人调用 Coze 工作流：
    
    ```
    POST https://api.coze.cn/v1/workflow/run
    {
      "file_token": "xxx",
      "task": "视频拆解"
    }
    ```
    
4. Coze 处理视频
    
5. Coze 返回 Markdown
    
6. 机器人把 Markdown 写回飞书文档
    

### 用户看到

文档里自动出现：

- 视频分段
    
- 关键帧截图
    
- 文案总结
    
- 结构化内容
