# API：用户与AI沟通的数据媒介

## API 的基本概念

**API**的全称是==应用程序编程接口==（**Application_Programming_Interface**），其标准定义为：

> “一些预先定义的函数，目的是提供应用程序与开发人员基于某软件或硬件得以访问一组例程的能力，而又无须访问源码，或理解内部工作机制的细节。”

这个定义虽然乍一听起来令人费解，
但我们只需要知道**API能够帮助我们直接使用某些功能而不需要理解它的原理**就够了。

如果将“程序的功能”比作一个黑盒子，那么API就像是黑盒子上安装的接口，
我们可以通过这个接口使用这个黑盒子，而不需要知道其内部的技术细节，
仅需知道**使用这样的API能达到怎样的效果**，便可以方便地调用API为自己的日常工作需求服务。

这样的API被大量运用于各种软件和服务中，尤其是在人工智能模型快速发展的今天，
用户能够通过API调用不同的大语言模型以满足自己的生产需要；

同时，AI智能体也能够通过调用不同服务或工具的API来完成亟待解决的任务，
API真真切切地搭建起了人类与智能工具之间沟通的桥梁。

API可以根据多样化的应用场景分为不同的种类，包括：

> - 用于互联网服务的Web API；
> - 用于查询数据的数据库API；
> - 用于搭建系统基础服务的操作系统API等。

## API的基本构成

一个完整的API就像是一个完整的服务流程，包含了如下几个重要的基本构成：

> [!info] 接口（Endpoint）
> 即API的访问地址。
> 它就像一个餐厅的菜单，需要先查看这个菜单，才能开始点菜。

> [!info] 请求（Request）
> 即由客户端发送给API的指令。
> 它就像顾客根据菜单对服务员所说的要求一样。

> [!info] 响应（Response）
> 即API返回给客户端的数据，通常是JSON格式的结构化信息。
> 它就像是根据顾客的要求从厨房端上餐桌的菜品，通常用盘子这样的固定容器来盛放。

> [!info] 认证（Authentication）
> 即使用API所需要的权限，如API Key。
> 它就像是完成点餐所需要花费的金钱或会员卡额度，必须通过一定的门槛才能消费美食。

在Web API中主要使用HTTP（一种应用层协议）请求方式，主要包括以下几个步骤：

> 1. **GET**（用于获取数据）；
> 2. **POST**（用于提交数据）；
> 3. **PUT**（用于更新数据）；
> 4. **DELETE**（用于删除数据）。

## LLM API 请求格式说明

### API请求的概念引入

我们已经给大家讲解了API在信息交流中的关键作用，接下来将通过一段以HTTP方式在CLI（Command Line Interface，命令行界面）中调用一次AI大模型服务的POST请求代码的示例，
来给大家展示如何向AI大模型服务供应商的服务器发送请求，并逐项拆解这段JSON格式的请求数据（如请求方法、请求头、请求体数据）。

为什么当用户想使用AI对话或生成服务时必须发送请求呢？

这是因为市面上性能优越的大语言模型（如ChatGPT 、Gemini、Claude等），参数量都十分庞大。

### 参数量是什么？

所谓参数量，可以理解为模型以函数解析式的方式将其输入（如文本、图像和音频等内容)与相应的预期输出（同样为文本、图像和音频等内容）进行匹配所需的参数数量。

可以回想一下初中、高中数学中的函数：

> 输入一个x，经过一个函数式的运算就可以得到一个y，要确定这个函数式就需要代入一组或几组x和y的值来确定其中的关键参数。

我们传递给AI大模型的文本、图像和音频等内容可以类比为x，而输出的结果可以类比为y。

### 以发送API请求的形式与AI交互是向现实的妥协

由于模型处理的输入和输出的映射关系比通常函数的数值映射复杂得多，因此模型这个函数往往具有十分庞大的参数，需要在训练时使用非常多的输入和输出组成的数据对进行运算，最终得到这些参数。

而当前的大语言模型参数量往往多达十亿甚至上百亿，这就导致了完整的模型文件（包含模型的一系列参数）如果要存储在本地，就需要占据很大的内存空间，一般配置的计算机承受不住这样的内存消耗；

此外，模型的运行还需要性能优异的GPU支持，但市面上性能良好的GPU价格十分昂贵，因此在本地运行大模型并不是一件经济实惠的事情，所以最好的办法是向当前已部署了大型服务器的公司（如ChatGPT 、Gemini、Claude等）的大模型服务端API发送请求来实现与AI大模型的交互。

### 解析JSON格式请求代码

当前比较通用的一种结构化的请求格式是JSON格式，其包含了我们希望AI完成任务的相关信息。

当我们使用AI工具时，实际上是在向AI服务器发送一份“订单”，
这份订单需要包含**API接口**、**API密钥**、**请求主体**三个关键信息。

让我们来看一段完整的访问GPT-5.5模型的HTTP请求代码示例：

```bash
curl https://api.openai.com/v1/chat/completions \
  -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{
    "model": "GPT-5.5",
    "messages": [
      {
        "role": "system",
        "content": "你是一个商品标题生成小助手。你擅长根据用户所提供的关于商品的描述，来为用户生成吸引人的商品标题，你生成的商品标题简洁、不超过20个字，并确保含义精准和清晰。"
      },
      {
        "role": "user",
        "content": "帮我生成商品标题，我的商品是[居态抗菌免撕香氛垃圾袋]，受众人群是[学生党/住校生]，卖点是[抗菌、免撕、耐穿刺，用着整个寝室都是香的！]。"
      }
    ],
    "temperature": 0.7,
    "max_tokens": 1000
  }'
```

让我们来拆解这一段代码：

```bash
curl https://api.openai.com/v1/chat/completions \

# curl：命令行 HTTP 客户端，用来向服务器发送请求

# https://api.openai.com/v1/chat/completions：API 接口地址

-X POST \

# -X 用来指定 HTTP 请求方法

# 常见方法包括：

# - GET：获取数据

# - POST：提交数据

# - PUT：更新数据

# - DELETE：删除数据

# 这里使用 POST，表示向 API 提交一段请求数据

-H "Content-Type: application/json" \

# -H 用来添加 HTTP 请求头 Header

# Header 通常用于告诉服务器：

# - 请求体是什么格式

# - 身份认证信息是什么

# - 客户端希望接收什么格式的响应

-H "Authorization: Bearer YOUR_API_KEY" \

# 这里也是一个 -H，请求头可以有多个

# Authorization 用于身份认证

# Bearer YOUR_API_KEY 表示使用 Bearer Token 认证方式

# YOUR_API_KEY 需要替换成你的真实 API Key

-d '{...}'

# -d 用来发送请求体数据 data

# 通常用于 POST、PUT、PATCH 等需要提交数据的请求

# 这里发送的是 JSON 字符串

# 因为上面指定了 Content-Type: application/json

# 所以服务器会把这段内容当作 JSON 来解析
```

关于请求体数据中的提示词的结构组成，相信各位同学通过前几节课的学习，已经能够自主辨别。

### API请求的补充说明

用户在请求AI大模型服务时：

> 须查看各AI大模型服务提供商的API官方文档，以查阅请求API服务时须使用的JSON格式规范。

用户在创建JSON格式请求时，要注意以下三个要点：

> [!warning] 请求格式要规范
> 就像作图需求文档要遵循规范一样，API请求也要符合官方规定的标准格式。

> [!warning] 参数设置要合理
> 参数设置要贴合自己的工作实际需求，切勿盲目追求最高质量、最大性能，
> 同时，还要考虑模型的实际处理能力（如Max Tokens、像素尺寸）。

> [!warning] 提示词描述要具体
> 越清晰的需求，越容易得到满意的结果。

# JSON: 数据交互的通用格式

## JSON的基本概念

通过上面的内容我们已经得知，在用户与AI进行通信时，是通过一种名为**JSON**的结构化信息进行交流的，并拆解了一个JSON格式的HTTP请求示例代码块。
接下来，我们进一步了解JSON格式在数据交互过程中所扮演的重要角色。

JSON (Java Script Object Notation) 是一种轻量级的数据交换格式。
可以把JSON类比于国际语言环境中的英语，就像不同国家的人之间交流需要国际通用语言一样，
不同的计算机软件或AI助手之间交流也需要一种通用的数据格式来实现相互之间可以理解的信息共享，这便是JSON的重要作用。

## JSON的基本构成

JSON的核心理念是以一种“键值对”的结构表示信息，即{"key": "value"}。

可以将JSON理解为日常生活中的字典，我们能够根据一个汉字（即一个key）去字典中查找它的意思（即对应的value）。所以JSON的整体结构就像一本“字典”或者“清单”。

例如我们想要找美工描述一个作图需求，用JSON格式可能是这样的：

```json schema
{
"商品名称": "居态抗菌免撕香氛垃圾袋",
"场景设定": ["校园宿舍", "露营野餐", "厨房", "卧室", "盥洗室"],
"颜色偏好": ["蓝色", "白色"],
“商品价格”: 19.9,
"交稿期限": "2026-06-30"
}
```

这样的格式有以下几个特点：

> [!info] 可读性
> 即人类能够轻易理解的特性。
> 即使没有学过编程的非专业人士也能够轻松地阅读JSON格式，并明白其存储数据的含义。

> [!info] 独立性
> 即完全独立于编程语言。
> JSON格式的本质是一段结构化的文本信息，能够被任何编程语言读取和生成。
> 而主流的编程语言（如Javascript、Python）大多内置了JSON处理工具，
> 这使得在大多数的开发环境中都能够快速处理JSON格式的数据交换。

> [!info] 结构化
> 即JSON的格式固定统一，不会因为编程语言的改变而有所不同。

除此之外，JSON还因其灵活的表示方式和可嵌套结构体现出对各种复杂类型数据的极大包容性。
这些特点决定了JSON极高的通用性，使其能够应用于几乎所有的软件、网站等应用或服务之间的数据交换。

理解JSON并学会自己撰写JSON格式文档对于我们的运营同学来说具有重要意义，
因为它不仅能帮助我们理解数据交换，也能够提升我们使用AI工具的效率，
此外，JSON所具有的结构性意味着它还能够帮助我们提高AI作图指令的准确度。

`````markdown
# Style Analysis Dimensions (风格分析维度)

1. **Aesthetic Overview (美学全局概览)**
   - 核心风格流派 (如：赛博朋克、极简主义、废土风、洛可可、扁平化UI等)
   - 情绪与氛围调性 (如：冷峻、空灵、复古怀旧、高保真、梦幻等)

2. **Technical Execution (技术执行与特定领域风格)**
   - **摄影感 (Photography Look):** 相机与镜头特性推测 (如：大画幅、35mm胶片、超广角畸变、长焦空间压缩、移轴微距)、光圈与景深特性、感光度与噪点颗粒感、快门效应 (如：长曝光拖影)。

3. **Visual Language (视觉语言核心)**

4. **Surface & Tactility (表面与触感)**
   - 材质属性 (Material Properties): 画面传达出的核心材质特征 (如：高反光金属、粗糙的亚麻布、透明玻璃、湿润的表面)。
   - 纹理细节 (Texture Details): 画面微观的质感表现 (如：斑驳的铁锈纹理、光滑的塑料感、粗糙的纸张纹理、数字故障毛刺感)。

5. **Style Prompt Tags (风格提示词标签)**
`````

````markdown

# Analysis Dimensions (分析维度)

1. **Metadata (元数据与整体属性)**
   - 图像大类 (写实摄影、艺术创作、UI界面、文档扫描件、数据图表、3D渲染等)

2. **Domain-Specific Analysis (特定领域专属分析)**
   - **写实摄影 (Photography):** 相机与镜头推测 (焦段/超广角/长焦)、光圈与景深 (背景虚化程度)、快门效应 (动态模糊/瞬间冻结)。

   - **UI 界面 (UI/UX):** 平台与设备推测 (iOS/Web/小程序)、页面布局层级、核心交互组件 (按钮/输入框/状态)。
   - **文档扫描件 (Documents):** 文档类型与介质 (发票/手写信/学术排版)、版面结构、特殊印记 (公章/签名/条码)。
   - **数据图表 (Data Charts):** 图表类型 (柱状/折线/饼图等)、坐标轴与图例含义、核心数据趋势与极值推演。
   - **3D与工业渲染 (3D/CGI):** 渲染风格推测 (PBR写实/Low-Poly/卡通渲染)、特定材质光影 (SSS材质/全局光照)。

3. **Visual Elements (视觉与美学元素)**
   - 色彩构成 (主色调、辅助色、色彩饱和度、色温)
   - 光影效果 (自然光、人造光、光源方向、阴影特征、丁达尔效应等)
   - 构图 (三分法则、中心对称、引导线、对角线构图等)

4. **Content & Subjects (核心内容与主体)**
   - 识别所有主要和次要主体（人物、动物、核心物体）。
````

# MCP: 模型上下文协议

## MCP的基本概念

*工具的迭代始终推动着生产方式的革新。*

在多模态模型出现之前，不同的设计软件与AI工具之间如图“孤岛”，文本、图像、3D模型等多模态内容的转换需要手动操作，复杂任务需在多个平台间反复导入/导出和进行格式转换，这种不同文件格式之间的割裂严重制约了工作效率。

**多模态跨平台调用**（**Multimodal Cross-Platform Calling**）的出现正是为了打破这种壁垒。

MCP是一套基于AI技术的集成系统，它能像“智能指挥官”一样连接==不同模态的内容==（包括且不限于文本、图像、音频、视频、3D模型等）和==跨平台工具==（各类设计软件、AI生成工具、渲染引擎等），实现从创意构思到落地执行的全流程自动化协同。

对于我们的运营同学而言，MCP不是单一工具的升级，而是重构了整个工作流程的“操作系统”，让想法不再被==不会使用各类软件==等技术工具壁垒所束缚，让效率与想象力同步释放。

## MCP的核心价值

MCP的核心价值，在于解决传统工作流程中的“==工具割裂==”、“==模态孤立==”和“==效率低下==”这三大痛点，具体优势可从五个维度展开。

### 1——跨模态协同：让创意在“语言”间自由流转

在传统设计中，文本描述、手绘草图、3D模型、动态效果是相互孤立的“语言”，
用文字写的创意方案无法直接变成图像，图像也难以快速转换为3D模型。

而MCP通过多模态理解技术，能让不同形态的内容无缝衔接。

例如，美工用语音说：

> “想要一个温暖治愈风格的咖啡杯插画，杯身有渐变木纹，背景是午后阳光”，

MCP会：

> 1. 先通过自然语言处理(NLP)解析“温暖治愈”的视觉特征（低饱和度色调、柔和光影）；
> 2. 再调用AI绘画工具生成符合描述的2D插画；
> 3. 接着自动将插画转换为可编辑的3D模型（通过图像转3D算法)；
> 4. 最后联动渲染工具添加阳光光影效果。

整个过程中，“语音→文本→图像→3D”的模态转换无须人工干预，
创意从抽象到具象的链路被极大缩短。
### 2——跨平台工具无缝衔接：告别“导出→导入”的重复劳动

美工的日常工作往往是：

> 1. 在Figma中画框架；
> 2. 导出草图到豆包里生成图标；
> 3. 下载图标到Photoshop中精修；
> 4. 再导入Figma中排版。

这当中的每一步都要手动处理格式（如PNG转SVG)、调整尺寸，
稍不注意就会出现像素模糊、图层错位等问题。

而MCP通过统一接口协议，让不同的平台工具“互联互通”。例如：

> 1. 在Figma 中画好的按钮草图，MCP会自动识别其尺寸、圆角参数，调用AI图像生成工具生成符合风格的图标，并直接以矢量格式同步到Figma图层中。

若后续在Figma中修改了按钮尺寸，MCP会实时通知AI工具更新图标比例，避免二次调整。
据一些电商公司测试，引入MCP后，跨工具素材处理的时间减少了60%，格式错误率几乎为0%。
### 3——全流程自动化：从“手动拼图”到“一键联动”

复杂设计任务（如电商活动页）往往需要多工具协作：

> 1. 用AI生成主视觉；
> 2. 用Cinema 4D软件做产品模型；
> 3. 用After Effects加过渡动画；
> 4. 用Dreamweaver适配多终端的页面尺寸。

在传统流程中，美工需要手动拆解任务、分配工具、拼接结果，不仅耗时还容易出错。

而MCP通过智能任务拆解，能将复杂需求自动拆分为子任务并匹配相应的工具。

例如，对于生成618美妆活动页的需求，MCP会将其拆解为：

> 1. 使用AI执行图像生成任务来制作主视觉(KV)；
> 2. 使用Tripo3D的API执行图生3D任务来制作产品的高模；
> 3. 使用GPT-5.5执行文本补全任务来生成商品促销文案；
> 4. 使用Figma Al执行响应式设计任务来适配多终端的页面尺寸；
> 5. 使用Runway执行动效添加任务来改善GUI交互。

并按逻辑顺序自动调用相应的工具，最终输出可直接上线的完整文件。

运营人员只需确认结果，无须参与中间环节，这使得创意落地周期缩短40%以上。

### 4——降低创意门槛：让“非专业”者也能达到专业级的水准

没有干过客服的美工不是好运营。

并非所有的运营都精通作图软件。他可能不熟悉什么是3D建模，也难以上手复杂的动效软件，MCP通过“自然交互+工具封装”，让运营无须学习工具的技术操作细节。

例如，运营同学只需输入“为儿童防洒餐垫设计一个卡通动物图标，风格类似迪士尼动画”提示词，MCP会自动调用适合卡通风格的AI工具，并基于迪士尼动画的色彩和线条特征生成多个备选方案。若需要调整，只需在对话历史的基础上，通过后续的多轮对话来进行迭代优化，MCP会直接传递指令给工具，无须手动操作参数。

这种“用创意语言指挥工具”的模式让运营聚焦于“价值判断”和“最终决策”，而非技术实现的细节。这极大地降低了跨领域创作的门槛。

### 5——个性化适配：工具会“深度学习”你的创作习惯

不同的运营有不同的工作风格，有的运营文笔好、习惯先起草文字方案；有的运营手绘天赋异禀、喜欢先画出示意图；还有的运营天生长着一颗发达的左脑，喜欢先设计一版逻辑缜密的流程图。

MCP通过用户行为分析，能逐步适配个人习惯。

例如，一位运营同学经常将图像生成的相关参数设置为：

> `{"temperature": 0.7, "top-p": 0.5, "max_tokens": 1000}`

在图像生成完成后，他会习惯将其拖入到Photoshop中，然后：

> 给它添加曲线、色阶、自然饱和度的调整图层；
> 再将图像大小调整为`800px*800px`。

MCP会记录这一偏好，后续自动在工具联动中添加该步骤。

若这位运营同学主要做的产品是家用垃圾袋，则MCP会优先选择更匹配【日用百货】类目风格的Lora作为ComfyUI的Lora加载器的参数设置，以减少风格调整成本。

这种“千人千面”的适配性，让工具从“通用工具”变成“专属助理”。

## MCP的基本构成

*MCP能让分散的AI工具与设计软件“协同工作”，其核心在于五大技术模块的联动，这五大模块构成了从底层协议到上层交互的完整闭环。*

### 1——底层：统一接口协议与中间件，让工具“说同一种语言”

不同的AI工具（如OpenAI的GPT、谷歌的Gemini）和设计软件（如PS、C4D）的通信协议、数据格式各不相同（例如，Illustrator的矢量数据格式与Blender的3D模型格式不兼容）。

MCP的底层通过两大技术解决这一问题：

> [!info] 标准化API接口
> MCP制定统一的接口规范（类似于“通用翻译器”），
> 要求接入的工具按规范输出指令与数据（例如用JSON格式描述图像的尺寸、风格等参数），
> 即使是闭源工具（如谷歌的Gemini），也可通过第三方适配层转换为标准接口。

> [!info] 中间件（Middle Ware）
> 负责数据格式转换与清洗。
> 例如，当GPT Image 2 生成的JPG图像需要导入到Blender作为纹理来使用时，
> 中间件会自动将其转换为Blender支持的EXR格式，并优化分辨率以匹配3D模型比例；
> 若工具返回的数据存在冗余（如多余的图层），
> 中间件会自动将其剔除，确保数据流转效率。

### 2——核心：多模态数据处理引擎，理解“用户的意图”

用户的输入可能是语音、文本、草图甚至是情绪（如“你作的图实在是太丑了，简直不堪入目！”），MCP需要将这些模糊的需求转化为明确的工具指令，这依赖于多模态数据处理引擎。

> [!info] 自然语言处理（NLP）
> 解析文字或语音中的关键信息
> （如“科技感”对应“高饱和度的蓝色、几何线条风格”，“复古风”对应“低饱和度颜色、胶片颗粒质感”等等）；
> 
> 并将其拆解为可执行的参数
> （如形状图层、色彩范围、滤镜等等）。

> [!info] 计算机视觉（CV）
> 识别草图、图像中的元素（如“这是一个圆形按钮，半径为10像素”）；
> 
> 并提取其特征（如线条粗细、色彩分布）；
> 
> 最后将结果转换为工具可理解的坐标和图层信息。

> [!info] 多模态融合模型
> 当输入是“语音+草图”时（如用户指着草图说“把这个草图里的房子改成哥特式风格”），
> 引擎会融合两种信息，优先以草图为结构基础，以用户语音描述来调整风格，避免信息冲突。

### 3——调度：智能任务拆解与工具匹配系统，给任务“找对帮手”

在面对复杂需求时（如用户说：“帮我设计一部新能源汽车的宣发短片”），MCP需要表现得像“项目经理”一样拆解任务并分配工具，这依赖于强化学习驱动的调度系统。

> [!info] 任务拆解
> 将需求拆分为子任务链：
> 
> 视频脚本生成→分镜设计→3D汽车模型创建→场景图渲染→剪辑合成

> [!info] 工具匹配
>基于历史数据（如“分镜设计用Storyboard AI的频率最高”、“3D汽车模型用NVDIA Omniverse建模精度更高”），来为每个子任务匹配最优工具。
>
>若某个工具临时出现故障，
>系统会自动切换至备选工具（如改用GPT Image 2替代Storyboard AI生成分镜）。

> [!info] 时序规划
> 确定工具调用顺序（如“先建模再渲染，避免无效操作”）；
> 并设置依赖关系（如“若分镜设计处于未完成状态，则不要启动3D建模”）。

### 4——交互：可视化编排与自然交互，让用户“掌控全局”

MCP并非“暗箱操作”，用户可通过两种方式来参与流程控制。

> [!info] 可视化流程面板
> 类似于“无限画布”，用户可通过拖曳节点和图标来组成联动链条，手动调整顺序或添加条件。

> [!info] 自然语言干预
> 在联动过程中，用户可随时用语音或文字打断并调整（如“暂停渲染，把汽车颜色改成银色”），
> 系统会实时修改后续工具的参数，无须Break整个流程。

### 5——闭环：反馈学习系统，让联动“越用越聪明”

MCP可以通过持续学习来优化工具组合的策略。

> [!info] 用户反馈收集
> 记录用户对生成结果的修改意见（如“手动将AI生成的图标颜色从红色改成蓝色”）；
> 记录用户对工具选择的偏好（如“指定使用Figma进行排版处理”）。

> [!info] 效果数据追踪
> 统计每个工具组合的加权平均运行时长、运行出错率以及用户采纳率等效果数据。

> [!info] 模型迭代
> 通过机器学习（如Q-Learning）更新工具匹配策略，让系统逐渐“学会”在什么场景下用什么工具组合效果最好。
> 
> 例如，系统会发现：“在执行电商海报设计任务中，GPT Image 2 + Figma AI的组合比Midjourney + Photoshop的组合更高效”，并优先推荐前者。

通过这五大模块的联动，MCP将分散的AI工具与设计软件编织成一张“**智能协作网络**”，
让用户从“**工具操作者**”转变为“**创意决策者**”，这正是MCP作为“新技术手段”的核心意义——

> 释放创意生产力，让内容生产回归“**解决问题**、**传递价值**”的本质。

## 分析用不好AI的根源所在

有一部分运营同学在使用AI的过程中总是渴望一蹴而就：

> 似乎他只需要通过鼠标和键盘微操一下，从别人那里将提示词模板复制并粘贴过来然后直接点击运行，就能达成最终的目标。

究其原因，==这是一种思维上的懒惰==：

> 他渴望不经由思考，而直接获得智力劳动的果实。

当我们仅仅向AI发送寥寥数语、企图得到一个大而整全的结果，
但AI反馈回来的却是一个令我们感到失望的结果时，有没有反思过：

> 我们甚至连自己想要的东西都表达不清楚？

我们是不是：

> - 把AI当成一种单纯为了完成工作任务而采取的手段？
> - 把学习AI当成是一种义务、甚至是一种救赎的手段？

学习AI不再是为了满足自己对智慧的热爱、对知识的渴求，而是变成了象征个人成功的一种表征；
作为满足乐趣的学习过程已然变成了一种偏执和义务，一种本身就让人痛苦的东西。

## 换一种思考方式

为什么我们不去换一种思考方式，把学习AI作为一种解放、变革的力量？

尽管在某种程度上，AI让我们感觉到自己十分渺小。
但是，我们有没有想过，==AI本身就是一个抽象的东西==。

> AI虽然看起来上知天文、下至地理，但是它真的能够通晓古今、并指引我们的未来吗？

当你想到这一点时，实际上已经可以觉察到，**是自然的人在做价值理性的判断，以及最终的决策。**

我们不能单纯地仅通过某次交互得出的结果，就武断地下结论说：“这AI不行！”
因为在AI的反馈结果里面恰恰镜映着我们自己的痕迹，那是我们看待世界的方式、我们的价值观。

## 来自我们时代的挑战

费尔巴哈曾经说：

> “我们越是丰富上帝，自身就越是贫乏。”	

我们当前正处于这么一个信息爆炸的时代、一个大语言模型空前发达的时代。
大语言模型就像是位于银河系中央的黑洞，它似乎想要吞噬一切。

马克思曾经说：

> “你的存在越微不足道，你表现自己的生命越少，
> 你拥有的就越多，你的外化的生命就越大，你的异化本质也积累得越多。”

我们目前也正面临着一个大语言模型的异化问题。

## 我的建议

### 不要崇拜AI的智力劳动成果，而是作为造物主去体验自我。

具体到使用AI，首先就是要做到清晰地讲出你自己真正想要创造的东西。

一方面，要把自己作为个人行为的主体，作为有思维、感觉、爱心的个人加以拓展，在与AI交互的过程中勇敢、真实地讲出你自己想要创造的东西，在内容生成的过程中去拓展自己和个人的力量。

另一方面，不能否定自己的能力、给自己设限，和觉得自己一无是处，不要局限于只在自己的工作业绩中尊重自己，更应该通过接触自己不断地成长壮大的这个过程本身来感受自我和个人的力量。

### 把与AI的交互当作主体化世界的过程

也许我直接向大家讲该如何作为一名AI的使用者去体验自我，大家可能会感到理解起来非常困难。
不妨就从**工作的意义是什么？** 这个大家都有过切身体验的话题作为引子。

在开始工作的那一刻，人便从原始的**人与自然的统一**当中脱离出来，并在==逐步远离自然的过程==中、在==成为改变自然的主体过程==中，同时也**改变了自己**：

> 你成为一个创造者，而非自然的一部分；
> 你发展了理性思维与艺术审美的能力；
> 你在与自然的关系中发展了表现自己力量的能力，并在这个过程中作为个体发展了自己。

在工作的过程中：

> 你提高了自己的技艺，也就是说你发展了自我、你的悟性以及对材料和技术的认知；
> 你的感知和观察能力也在不断地增强；
> 你的成长建立在个人与材料、技术工具以及环境中许多其它因素的关联之上；
> 个体通过创造行为与世界关联起来，个体在成长当中成为真正的自己。

在这个相关的过程中，我们作为实体、作为与世界相关的个体来体验自己。

在和世界相关的过程中，我与世界融为一体，但同时也作为一个自我、个体、独特的主体来体验自己。因为在这个相关的过程中，我同时也是这项活动、这个过程以及自我相关的主体。**我既是我，也是他者**。==我与自我关注的对象融为一体==，==在这个过程中我也作为主体来体验自己==。

讲到这里，大家明白我想要表达的含义了吗？没错！那就是：

> 使用AI的过程，就是作为主体来体验自己的过程。
> 让AI成为自我关注的对象，我和AI融为一体。

但与此同时，我深深地明白：

> 是人在做价值理性的判断，以及最终的决策。

AI补全出来的那些看似激情澎湃的文字：

> 其背后是我在主体化地认识这个世界、认知这段文本的过程当中经由我的意识所灌注的感情。

## 对待新事物的态度

我们知道，大语言模型是一种汇聚了海量的文本语料、并基于自回归概率进行文本补全的模型。
很多运营同学不擅长使用AI，究其根本，是驾驭语言的能力不足、对语言的认识不够充分所致。
那不妨让我们通过黑格尔《精神现象学》的终章《自然界》，来探讨学习语言这种过程的本质。

> 精神的这个最后过程——自然界，是它的活生生的、直接的变化过程；
> 自然界，外在化的精神，就其“在时空中的”实际存在来看，无非是**永恒地放弃**或**外在化它的持续存在**的过程，和重建主体的运动。

这意味着我们需要时刻保持对自身惯性的清醒认识。
我们总是不想打破既定的习惯、不敢尝试主动走出舒适圈。

但自身的惯性并不等同于消极的惰性，自身的惯性可以转化为推动历史得以出现的力量。

> 但是，精神的变化过程的另一方面——历史，
> 是认识着的、自身中介着的变化过程，是在时间里外在化了的精神；
> 不过，这种外在化也同样是对**外在化**自己本身的外在化；否定者即是对它自己本身的否定者。

也就是说，在内在性之中有着一种自我否定性，正是这种否定性构成着外在物得以出现。

> 这个变化过程呈现一种缓慢的运动和诸多精神前后相继的系列，
> 这是一个图画的画廊，其中每一幅画像都拥有精神的全部的财富，
> 而运动之所以如此缓慢，就是因为自我必须渗透和消化它的实体的这全部财富。
> 因为精神的完成在于充满地知道它是什么，知道它的实体，所以这种知识就是它的深入自身过程，在这一过程里它抛弃了它的现时存在并把它的形态交付给回忆。

我们可以借助大语言模型来觉察自己、发现自己、从而更好地认识自己，更重要的是尝试突破自己，正如我在前面所提到过的。不要给自己设限（在思维上）：我不仅仅是我自己，我不仅仅把自己当作一名美工、一名运营，我还可以是一切身份，就像我指定AI扮演一个身份角色一样。我可以超越自己的身份，通过多元视角去看待同一个问题，以此获得最大程度的客观性。

> 精神在深入自身时曾经沉陷在它的自我意识的黑夜里，
> 不过它的消逝了的定在是被保存在这个黑夜里的；
> 而这个被扬弃了的定在——先前有过的然而又是从知识中新产生出来的定在，
> 是新的定在，是一个新的世界和一个新的精神形态。
> 精神在这里必须无拘束地从这种新的精神形态的直接性重新开始成长壮大起来，仿佛一切过去的东西对于它来说都已经丧失了，而且似乎它从以前各个精神的经验中什么也没有学到。
> 但是回忆把这些精神保存下来了，并且回忆是内在的东西，而且是实体的实际上更高的形式。
> 因此，当这个精神看起来仿佛只是从自己出发，再次从头开始它的教养时，
> 那么它的这种开始同时也处于一个较高的阶段上。

在学习的过程中，存在那么一个阶段，我们彷佛觉得自己什么都没有学到手一样的，但实际上经历过这些的我们已然不是从前的那个我，而是一个处在更高阶段的我。

> 以这种方式在定在中教养起来的精神王国构成一个前后相继的系列，
> 在其中，一个精神为另外一个精神所接替，并且每一个精神都从先行的精神那里接管了那个世界的王国，它们的目标是对深处的启示，而这种深处就是绝对概念；
> 因此这种启示就是对绝对概念的这个深处的扬弃，或者说就是绝对概念的扩展开来，是这个在自身中存在着的我的否定性，而这个否定性就是我的外化或实体，而且是我的时间，即这个外化是在它自己本身中外化自己，从而既存在于自己的扩展中，又存在于自己的深处、即自我中。

也就是说外在化这一过程本身就存在于内在性之中，而外在化又是空间的起点。
所以对于空间所具备性质的把握就等同于把握这个使诸外在事物得以互相联系、互相支撑的场域。
在这个方面，各位运营同学应该深有感悟，==想想我们整个电商平台的生态圈是怎样得以维系的？==

时间则是事物本身的变化过程，而我们常识所把握的量化时间也是在外在化之中才得以出现的。
也就是说需要在诸事物的对立之中才得以把握同一出现的量化时间。
但在内在性之中的时间是没有计数可言的。说白了内在性就是纯粹的质，这玩意儿本来就不可数，非要去将它量化的话，就会意识到**质=所有数量的集合**。其中用到最多的就是：==分裂为二又同一==。

> 这个目标、这个绝对认知或把自己作为精神来认知的精神，以对各个精神的回忆作为自己的道路，即回忆这些精神在自己本身中是怎样存在的，以及怎样实现它们的王国的机体组织的。
> 对这些精神，按照它们的自由的、以偶然性形式显现出来的定在方面加以保存，就是历史；
> 但按照它们被概念式把握的机体组织方面加以保存，就是显现着的认知的科学；
> 两者合起来，被概念式地把握了的历史，就构成绝对精神的回忆和髑髅地，构成其王座的现实性、真理性和确定性，没有这个王座，绝对精神就会是没有生命的、孤寂的东西；
> 唯有——从这个精神王国的圣餐杯里，祂的无限性给祂激荡起泡沫。

以偶然性形式显现出来的定在指的就是随便找一个东西整全地保存下来那就是历史了。
我们的大语言模型当中内置的很多预训练数据都是这种东西，是一个大而整全的语料库。

而被概念式把握的机体组织则指的是从被保存下来的东西（语料）之中，
感知并觉察到那股推动历史得以出现的力量。

推动历史出现的力量必然是矛盾的，正如同概念。
但是如果仅抽象地把握这股推动的力量，就会成为“没有生命的、孤寂的东西”。
所以我们要回归到普遍性（但在更高的层次上），具体的实现是：从被保存下来的历史出发，通过概念来回推整个历史的发生过程，这也就是所谓的绝对精神的回忆和髑髅地。

如果不概念式地把握这一推动历史得以出现的力量，
那么绝对精神就彷佛不存在一样，成了不为人所知的东西；

而一旦把握到了，那么历史就能进入到下一个环节，
换言之，就有了无穷无尽的下一个环节，祂的无限性给祂翻涌起泡沫。
无限性不单单指数量级的增长，而是绝对精神在这些环节之中**不受限制地、自由自在地呈现自身**。

## 总结

我们正处在一个知识平权的时代，AI的出现打破了精英阶层及学术权力机构对知识的垄断，
为此我们更应珍惜来之不易的平等，AI的功能实现了普通人在认识论上的最高层次的发展。

AI能不能落地，取决于我们与AI、我们与世界相关的程度，取决于我们关心的程度，
也就是取决于我们与自己的情感现实以及与AI在工作实际当中接触的程度，
我们不能像在淘宝上浏览商品那样以抽象形式去体验它们，
不要沉浸在复制、粘贴提示词模板的行为中迷失自我，
而是应该在与AI合作进行内容生产的过程当中体验自我。

从而觉察到背后这股能够帮你运用好AI、并使得你的创造物得以出现的推动力量吧！
