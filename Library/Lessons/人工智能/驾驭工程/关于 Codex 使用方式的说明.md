# 模型选择策略

Codex 支持多个模型，不同模型在能力、速度和成本方面存在不同的取舍。作为使用者，应根据任务类型选择合适的模型。

## GPT-5.6系列模型的三个层级

如下表所示，Codex 模型大致可以分为轻量级、标准级和高级三个层级。

<p align=center>GPT-5.6系列模型的三个层级</p>

| 模型            | 层级  | 特点      | 适用任务                                     | 相对速度       | 相对成本 |
| ------------- | --- | ------- | ---------------------------------------- | ---------- | ---- |
| gpt-5.6-luna  | 轻量级 | 快速但能力有限 | 重复、批量、低风险任务                              | 快（秒级响应）    | 低    |
| gpt-5.6-terra | 标准级 | 性能和成本平衡 | 大部分内容、整理、分析、普通判断                         | 中等（10～30秒） | 中等   |
| gpt-5.6-sol   | 高级  | 能力最强    | 复杂推理、长链路 Agent、架构设计、多模块重构、疑难 bug 修复、关键决策 | 慢（1～3分钟）   | 高    |

不需要每次都纠结选哪个模型，实用的选择策略是按任务复杂度分级。
* 默认用 gpt-5.6-terra：单文件修改、添加测试、简单重构、bug 修复——标准级模型足以应对这些任务。
* 简单任务用 gpt-5.6-luna：格式化代码、添加注释/文档、生成样板文件、简单的配置修改——这些任务逻辑简单，轻量级模型的产出质量和标准级模型几乎没有差别，但速度快很多。
* 复杂任务用 gpt-5.6-sol：跨多个模块的重构、需要深度理解业务逻辑的实现、复杂的算法实现、涉及架构决策的改动——这些任务值得多花时间和成本来获得更好的结果。

应先快速产出框架以确认方向正确，再精细打磨具体实现。这比一开始就用高级模型从头生成更高效，也更容易控制方向——如果骨架结构本身存在问题，那么及早发现比修改了大量细节后才返工的成本更低。

## 成本与速度的实践指南

实用的成本参考（以当前定价为准，实际价格可能略有调整）如下。

*   **简单任务**：几百到几千 token，成本极低（通常不到1分钱）。
*   **中等任务**：几千到几万 token（通常几分钱到几角钱）。
*   **复杂任务**：几万到几十万 token（通常几角钱到几块钱）。

以下是一些控制成本的实用技巧：

*   **避免重复执行**：如果 Codex 已经完成了一个任务，那么请你不要因为“效果不够好”就反复让它重做。使用迭代指令来优化，比完全重新执行更节省 token。
*   **控制上下文长度**：过长的任务描述会消耗更多 token。保持指令简洁但完整，不要在指令中粘贴大段代码（Codex 自己能读文件）。
*   **使用轻量级模型处理简单任务**：处理格式化代码、添加注释等简单任务时，使用轻量级模型能大幅节省成本。
*   **监控用量**：定期查看 API 用量面板，了解消费模式，及时发现异常情况。

模型的响应速度会影响你的工作节奏：

- 轻量级模型通常在几秒内开始输出，适合需要快速迭代的场景；
- 标准级模型的启动分析可能需要 10～30 秒，随后持续输出——这段时间并没有浪费，Codex 会分析项目结构、理解代码依赖、规划执行步骤；
- 高级模型处理复杂任务时可能需要进行 1～3 分钟的深度分析，但产出质量通常更高。

一个实用的建议：

> 不要盯着进度条看。

Codex 工作时，你可以做些其他事情（查邮件、看文档），等它完成后再回来审查结果。

# 安全边界：哪些任务不应交给 Codex

知道 Codex 能做什么很重要，知道它不应该做什么更重要。
不恰当地使用 Codex 不仅可能导致出现错误的结果，还可能带来安全风险。
下面将列出五类高风险任务，并给出应对建议。
这不是在限制你使用 Codex，而是在帮你建立安全边界。

## 五类高风险任务

下面列出的高风险任务，并不意味着 Codex 在这些领域完全无法参与，而是强调人类工程师不应把决策权或高风险执行权直接交给它。

风险越高，越需要将任务拆解：

> 让 Codex 参与分析、生成草案、补充检查或准备脚本，
> 并将审批、审查和最终执行交由人类工程师完成。

基于这一原则，以下五类任务尤其需要我们保持边界意识。

---
###  任务一：涉及敏感数据的操作

涉及数据库密码、API 密钥、用户个人信息和商业机密时，先按组织的数据分类标准、工作区策略和最小权限原则进行处理。能用脱敏样例、测试夹具和代码结构说明的问题，就不要在任务中使用真实敏感数据。

具体的数据使用、保留与治理边界，应以当前官方说明和所在组织的配置为准。从工程实践角度来看，不要把真实生产数据放在默认协作路径中，通常比事后追问“这次是否越界”更为稳妥。

---
### 任务二：生产环境的直接变更

Codex 能执行 shell 命令，这意味着它有能力操作远程服务器、数据库和容器编排系统。

但把生产环境的变更交给 AI 处理是极其危险的：

- 生产环境的很多操作是不可逆的（如删除数据、重启服务、修改配置），一旦出错影响范围可能很大；
- 生产环境的变更通常需要经过审批、预发布验证、灰度发布等流程——这些是 AI 无法替代的；
- 生产环境的变更还需要配套的监控告警和回滚预案——AI 不会设置这些；
- 如果 AI 的操作导致生产事故，那么责任怎么界定？

应对措施：

- Codex 可以生成变更脚本或 IaC（基础设施即代码），如 Terraform、Ansible playbook，但实际部署应由人类工程师在 staging 环境验证后手动执行；
- 需要修改环境配置时，应优先通过修改 IaC 来实现，而不是直接变更生产环境。

---
### 任务三：安全相关的代码

安全相关的代码需要经过严格的安全审计。

Codex 生成的安全相关的代码可能在功能上是正确的（token 能生成和验证、支付流程能跑通），
但可能潜藏细微的安全漏洞：

- 使用不安全的随机数生成器生成 token（应使用 secrets 模块而不是 random 模块）；
- JWT 的密钥硬编码在代码中（应从环境变量读取）；
- 文件上传没有校验文件类型和大小（可能导致恶意文件上传）；
- SQL 查询未采用参数化方式（可能导致 SQL 注入）；
- 权限检查不完整（比如只检查了用户是否登录，没检查用户是否有权限访问特定资源）。

这些问题难以通过功能测试发现：

> 功能测试只验证“正确路径”能够工作，而安全漏洞通常出现在“错误路径”中。

应对措施：

- Codex 可以生成初版代码，但必须由具备安全经验的工程师进行审查；
- 使用 SAST（静态应用程序安全测试）工具（如 Bandit、Semgrep）自动扫描安全问题；
- 关键安全模块（认证、授权、支付、加密）建议人工编写，Codex 辅助编写测试用例。

---
### 任务四：不可逆的破坏性操作

大规模删除或覆盖操作的恢复成本极高。

虽然 Codex 在执行破坏性操作前通常会请求确认，但风险仍然存在：

- 你可能在不完全理解影响范围的情况下批准了操作（“删除所有未使用的依赖包”看似无害，却可能删除了某个间接依赖）；
- 确认提示可能被你习惯性地点击“yes”（就像看到弹窗就点击“关闭”一样）；
- Codex 对“影响范围”的判断可能不完全准确。

应对措施：

- 执行前让 Codex 先列出会受影响的内容（启用 dry-run 模式）；
- 始终在 git 分支上操作，确认无误后再合并到主分支；
- 对于数据删除操作，先备份再操作；
- 对于 git push --force 这类操作，严禁 Codex 执行。

---
### 任务五：需要深度业务理解的决策

这类任务涉及业务规则、产品策略、技术路线等需要深入理解上下文才能做出的决策。

Codex 可以分析利弊（“实时推送的优点是……，缺点是……；轮询的优点是……，缺点是……”），但**最终决策权须归属人类工程师**。

应对措施：

- 让 Codex 提供分析而非结论——“列出三种方案的优缺点”比“帮我选一个方案”更合理；
- 最终决策权应始终掌握在人类手中。

## 分级信任原则

综合以上五类高风险任务，可以归纳出如下实用的分级信任原则。

*   **可完全信任（低风险、可快速审查后接受）**：格式化代码、添加注释和文档；编写单元测试和集成测试；简单的增删改查实现；原因明确的 bug 修复；依赖版本升级；代码翻译（不同语言之间的等价转换）。
*   **需要审查后使用**：业务逻辑实现（检查业务规则是否正确）；API 设计（检查接口是否合理、一致）；数据库迁移脚本（检查数据完整性）；性能优化（检查优化方向是否正确）；代码重构（检查是否有遗漏）。
*   **建议人工完成或深度审查**：安全相关的代码（认证、授权、加密、支付）；架构决策（技术选型、系统设计）；生产环境变更（部署、配置、数据迁移）；涉及敏感数据的处理；不可逆的破坏性操作。

这个分级信任原则并非固定不变：

> 随着经验的积累和对 Codex 信任度的提升，可以适当放宽某些类别的信任范围。

但在初期，保守一点没有坏处。

过度保守（什么都不让 AI 做）和过度信任（什么都让 AI 做）皆不可取，正确的做法是：

> 根据风险等级选择合适的使用方式——风险越高，人类的参与程度就应该越深。

# 可复用的工作流

随着使用 Codex 次数的增多，你会发现很多任务遵循相似的模式。
把每次都从零开始的过程固化为可复用的工作流，是进一步提升效率的关键。

让我们将这个提炼过程系统化。

## 三步法框架

每次使用 Codex 都围绕三个核心环节展开。这三个环节分别对应工程交付的“准备—执行—验收”三阶段，实质上构成一个“输入—处理—输出”的质量控制闭环，我们将其总结为“三步法框架”：

> 1. **准备上下文**：确保“输入”的质量——Codex 看到的信息越准确、越完整，产出越好。
> 2. **下达任务**： 定义“处理”的规格——做什么、怎么做、约束是什么。
> 3. **验收结果**： 检查“输出”的质量——是否正确、是否完整、是否符合规范。

在这个思维模型下，如果 Codex 的产出质量不理想，可按照如下三个环节逐一排查。

> - **上下文不足**：补充项目文档、优化文件结构、在指令中提供更多的背景信息。
> - **任务描述不清**：改写指令，增加具体的约束和验收标准。
> - **验收标准不够严格**：提高测试覆盖率、增加人工审查的深度。

掌握了这个排查思路，你就能快速定位并解决使用 Codex 时遇到的问题。

## 从手工到自动化

当你发现某个模式反复出现时，就可以考虑将其自动化。

Codex 提供了三个层次的自动化机制：

- **技能（Skill）**：如果某任务模式经常出现（比如“添加一个新的 CRUD 模块”），那么可以将其封装为 Skill。Skill 是可复用的任务模板，包含预定义的上下文加载策略、任务描述模板和验收步骤。触发 Skill 只需一句指令。

- **钩子（Hook）**：如果你希望在 Codex 执行的特定节点自动触发某些操作（比如每次写入文件后自动运行 formatter，每次 commit 前自动运行 lint），那么可以配置 Hook。Hook 是事件驱动的自动化机制。

- **子智能体（Subagent）**：如果某任务类型需要特定的处理流程（比如代码审查需要专门的审查策略、安全扫描需要专门的检测规则），那么可配置专门的子智能体。子智能体是角色化的自动化机制。

过早自动化是一个常见陷阱：有些工程师在用了几次 Codex 后就急着写 Skill 和 Hook，试图把一切流程都自动化。但因为没有足够的实践经验，他们设计的自动化流程往往不够灵活——能处理特定的任务模式，但稍有变化就失效。

因此，现在你需要记住的是：

> “好的工作流是从实践中提炼出来的。”

不要一开始就追求复杂的自动化，要先掌握基本的三步法框架。

等你积累了足够的经验后，就可以识别出哪些环节重复性高、哪些步骤可以自动化。到时候再引入 Skill 和 Hook，自然水到渠成。

# 课后总结

几个关键要点如下：

* **执行链路是五步闭环**：需求澄清 → 任务下达 → 执行观察 → 结果验证 → 代码提交。每一步都不能跳过，这不是建议，而是规范。这里特别强调结果验证——永远不要不经审查就接受 Codex 的产出。“信任但验证”不是口号，而是实践准则。
* **指令四要素**：做什么、在哪里、怎么做、怎么验。缺少其中任何一个，产出质量都会明显下降。
* **模型选择不必纠结**：轻量级模型快而简洁、标准级模型兼顾性能与成本、高级模型能力最强但速度慢。大多数任务使用标准级模型即可。成本在专业开发背景下几乎可以忽略。更重要的是任务描述的质量——一个好的指令配合标准级模型，远胜于一个差的指令配合高级模型。
* **知道安全边界比知道能力更重要**：涉及敏感数据的操作、生产环境的直接变更、安全相关的代码、不可逆的破坏性操作、需要深度业务理解的决策——这些不应该直接交由 Codex 处理。分级信任原则是你的安全护栏：常规代码信任，业务逻辑抽查，安全架构必审。初期保守一点，随着经验积累再逐步放宽。
* **三步法是可复用的框架**：准备上下文 → 下达任务 → 验收结果。这是高效使用 Codex 的基础。当你在实践中反复应用这个框架后，便能识别出哪些环节可以自动化——届时 Skill、Hook 等高级功能自然有了用武之地。

如果你在学完这节课后进行了实际操作（安装 Codex、执行三步法框架），那么你对 Codex 的使用深度就已经超过了大多数浅尝辄止的用户。接下来，我们将在此基础上不断深化和扩展——深入工作流的设计与优化，建立可复用的工作流模板和自动化机制，让 AI 应用从“手工活”变成“流水线”。
