# Markdown 负责记忆，HTML 负责展示：AI 时代新内容分工

从 Claude Code 的一次 HTML 热潮，看知识工作流真正该怎么拆层。

# 摘要

最近，Anthropic Claude Code 团队成员 Thariq Shihipar 发表了一篇关于 HTML 输出的文章，核心观点很直接：当 AI 能生成复杂计划、报告、原型和代码审查说明时，Markdown 的轻便优势不再总是占上风，HTML 反而能把信息组织得更清楚、更愿意被人打开阅读。

但这不等于 Markdown 该被淘汰。更实用的判断是： **Markdown 负责底层逻辑与记忆的纯净存储，HTML 负责对外的高密度交互与展示。** 一个像仓库，一个像展厅。仓库要干净，展厅要好逛。

# 结论

- Markdown 不该只被当成“文章格式”，它更像 AI 工作流里的\*\*底稿、账本和长期记忆\*\*。
- HTML 也不该只被理解成“网页开发”，它正在变成 AI 输出复杂信息时的\*\*展示层、交互层和协作层\*\*。
- 对普通知识工作者来说，最值得学的不是写前端，而是学会对 AI 说：\*\*先用 Markdown 保存逻辑，再用 HTML 做成能看、能点、能分享的页面。\*\*

# 一、真正的问题不是格式，而是“谁会真的读”

很多人第一次看到“用 HTML 替代 Markdown”这个说法，会本能地抗拒。

Markdown 多好啊。干净、轻、好复制、好进版本库。一个标题、几个列表、几段说明，基本就能把事情说清楚。过去几年，我们让 AI 写计划、写总结、写 PRD、写会议纪要，默认几乎都是 Markdown。

问题是，AI 变强之后，它写出来的东西也变大了。

一份实施计划可能有十几个模块，一份代码审查可能夹着 diff、风险等级、调用链和修复建议，一份研究报告可能横跨网页、代码库、聊天记录和历史版本。这个时候，Markdown 还在努力用列表套列表、表格套表格、ASCII 字符画流程图。

它不是不能表达，而是表达得太费眼睛。

Thariq 在原文里说，他越来越不愿意读超过 100 行的 Markdown。这个感受很真实。不是 Markdown 不好，而是当一份文档承担了太多展示任务，它就开始变成一堵文字墙。

这正是 HTML 的机会。

HTML 可以放图表、分栏、选项卡、锚点、按钮、交互控件。它能把“同样的信息”变成“更容易被人检查的信息”。而 AI 现在恰好能帮我们生成这些页面，不需要每个人都变成前端工程师。

关键变化在这里：过去我们选择 Markdown，是因为人要亲手编辑；现在很多内容已经由 AI 生成和修改，人更多是在阅读、判断、反馈。格式的重心，自然就从“好写”转向“好看懂”。

# 二、Markdown 没输，它更适合做底层存储

但如果因此得出“Markdown 已经过时”，那就走偏了。

在实用工作流里，Markdown 仍然有非常硬的价值：它适合存储那些需要长期稳定、反复调用、方便追踪的东西。

比如：一个项目的背景说明；一组提示词模板；某个客户的偏好；一份产品需求的变更记录；一套团队协作规则；一篇文章从选题到成稿的判断过程。

这些东西不一定要漂亮。相反，它们越朴素越好。

因为底层存储最怕三件事：看起来很炫但不好改；格式很复杂但不好追踪；页面很好看但 AI 下次读取时噪声太多。

Markdown 的优势，正好在这里：纯文本、结构清楚、方便 diff、方便放进知识库，也方便让 AI 在不同会话里重新读取。它像一本工作台账，不负责打动别人，只负责把事实、约束和判断记清楚。

所以，我更愿意把 Markdown 定义为： **AI 时代的记忆格式。**

它不是最终舞台，而是后台仓库。仓库里不用摆灯光，不用挂海报，但每个货架要标清楚，每次变更要找得到。

# 三、HTML 的新价值，是把 AI 的思考变成人能审查的界面

HTML 的价值，不是“比 Markdown 更高级”。它的价值是：当信息变复杂时，它能帮人重新进入循环。

这句话很重要。

AI 编程工具最让人兴奋的地方，是它可以一次性处理很多资料、代码和上下文。Claude Code 这类工具并不是单纯聊天窗口，官方文档把它描述为能读取代码库、编辑文件、运行命令，并和开发工具集成的 agentic coding tool。换句话说，它不仅会回答，还能推进事情。

可推进事情之后，另一个问题就来了：人怎么审查？

如果 AI 给你一份 300 行 Markdown 计划，你可能只扫标题；如果它给你一张页面，里面有模块图、风险标记、关键 diff、流程箭头、可展开说明和一键复制按钮，你更可能真的看完，并且做出判断。

这就是 HTML 对 AI 工作流的真正贡献： **它不是让内容变漂亮，而是让复杂内容重新可审查。**

比如代码审查。Markdown 可以写“这里可能有风险”，但 HTML 可以把风险按严重程度标色，把代码差异放在左边，把解释放在右边，把数据流画成图，再把修复建议折叠起来。读者不是被迫从头读到尾，而是可以按自己的问题进入文档。

再比如产品原型。Markdown 能描述“按钮点击后变成紫色并播放动画”，HTML 可以直接生成一个小页面，让你拖动滑块调整动画速度、颜色和缓动曲线，最后点一下复制参数。

这不是文档了。这是一个临时工作界面。

# 四、真正好用的分工：Markdown 管逻辑，HTML 管表达

把这件事落到日常工作里，可以形成一条很稳的链路：

第一步，把原始信息放进 Markdown。这里面包括事实、背景、约束、参考资料、用户反馈、你的判断。不要急着美化，先让它干净、可追踪。

第二步，让 AI 基于 Markdown 重新组织结构。让它提炼主线、拆出矛盾、列出关键取舍，必要时补一个“我应该重点审查什么”的清单。

第三步，让 AI 生成 HTML。不是为了炫技，而是为了把复杂信息变成更容易阅读的界面：对比表、流程图、交互原型、报告页、复盘页、评审页。

第四步，人来审查 HTML。看哪里不清楚，哪里判断不稳，哪里信息不够。审查完之后，不要只改页面，要把新的判断写回 Markdown。

最后一步，再进入下一轮。

这套循环的重点是： **HTML 是面向人的临时界面，Markdown 是面向长期协作的稳定记忆。**

很多团队的问题，恰恰是把这两层混在一起。把 Markdown 当展示页，就会越写越长，没人读。把 HTML 当长期记忆，就会越堆越复杂，难维护。分开之后，二者都轻松了。

# 五、普通人最应该借鉴的，不是 HTML，而是“可审查输出”

你可能会说：我不是程序员，也不用 Claude Code，这和我有什么关系？

关系很大。

如果你做公众号，可以让 AI 先用 Markdown 保存选题逻辑、资料来源、核心判断和段落结构，再让它生成一份 HTML 预览稿，里面放摘要区、重点句、对比图和适合手机阅读的排版。

如果你做培训，可以把课程大纲、学员问题、案例库保存在 Markdown，再让 AI 生成一个 HTML 学习页：左边是知识点，右边是练习题，底部是一键复制的复习清单。

如果你做产品，可以把需求和约束写进 Markdown，再让 AI 生成多个 HTML 方案并排对比：一个保守版、一个激进版、一个低成本版。你不需要先开 Figma，也不需要让团队在聊天记录里猜。

如果你做管理，可以把周报事实放进 Markdown，再让 AI 生成一页 HTML 管理看板：进展、风险、阻塞、需要拍板的事项，全都清清楚楚。

这些场景的共同点不是“会不会写代码”，而是“人能不能快速审查 AI 的输出”。

当 AI 输出变多，真正稀缺的不是内容，而是人的注意力。HTML 的作用，就是把注意力还给判断。

# 六、也别把 HTML 神化，它有自己的边界

HTML 很强，但它不是万能药。

它更耗 Token，生成也更慢；页面复杂以后，版本控制里的差异会很乱；如果你把每一份小笔记都做成 HTML，反而会增加维护成本。更现实的一点是，HTML 页面如果没有好的设计约束，也会变得花里胡哨，看似丰富，实际更难读。

所以，一个简单判断可以记住：

**需要长期保存、反复引用、方便追踪的，用 Markdown。需要让人快速理解、比较、操作和决策的，用 HTML。**

这也是我不太赞成“HTML 替代 Markdown”这个说法的原因。更准确的说法应该是：HTML 正在接管 AI 输出里的展示任务，而 Markdown 会继续承担底层记忆任务。

一个是保存判断，一个是呈现判断。一个负责不丢，一个负责看懂。

# 七、从今天开始，可以这样试三次

不用一上来就搞复杂模板，也不用先学前端。你只需要在下一次使用 AI 时，换三种说法。

第一种，用在资料整理后：

“请先把这些资料整理成 Markdown 版底稿，只保留事实、判断、引用来源和待确认问题，不要做花哨排版。”

第二种，用在需要给别人看时：

“基于这份 Markdown 底稿，生成一个单页 HTML 展示稿。请用对比表、流程图和重点提示块帮助读者快速理解，不要堆满文字。”

第三种，用在需要继续迭代时：

“我审查完这个 HTML 页面后，会给你反馈。请把新的判断更新回 Markdown 底稿，保留变更记录。”

这三句话背后，其实就是一个完整的 AI 内容工作流。

先记住，再展示；先沉淀，再交互；先把逻辑放干净，再把表达做清楚。

# 结尾

Markdown 和 HTML 的关系，不该是新旧之争，而该是前后台分工。

当 AI 还是一个回答问题的聊天窗口时，Markdown 足够好。可当 AI 开始生成计划、报告、原型、审查说明和临时工具时，我们需要一个更像界面的输出层。

所以，真正值得带走的不是“以后都用 HTML”。

而是这句话： **让 Markdown 负责记忆，让 HTML 负责被看见。**
